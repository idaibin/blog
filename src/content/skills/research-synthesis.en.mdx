Use this skill when the material is fragmented, sources disagree, or the user needs judgment rather than a plain summary. The skill turns sources into evidence, conclusions, disagreements, uncertainty, and action.

## Trigger conditions

Trigger this skill when the user asks things like:

- "Research whether this direction is worth building."
- "Turn these links into a conclusion."
- "These sources disagree; help me judge."
- "Give me a research brief I can make a decision from."
- "Do not just summarize. Tell me what to do next."

Do not trigger it when:

- There is only one source and a normal summary is enough.
- The user only wants translation or rewriting.
- The user clearly asks for creative writing instead of judgment.

## Required input

Before running the skill, make sure these are clear:

```text
Research question: ...
Audience: ...
Source materials: links / summaries / excerpts
Time range: ...
Output use: decision / documentation / product design / technical choice / writing
Constraints: length, language, citation requirements, conservative or opinionated tone
```

If the research question or source material is missing, ask first instead of proceeding.

## Workflow

### 1. Break down sources

For each source, extract:

- Source name
- Publication date
- Author or institution
- Main claim
- Supporting evidence
- Possible incentives or bias
- Whether it is a primary source

### 2. Cluster claims

Group the material into:

- Confirmed facts
- Product or technical changes
- Data or experimental findings
- Industry opinions
- Speculation and unverified claims
- Objections or risks

### 3. Calibrate confidence

Label each conclusion:

- High: multiple sources agree, with at least one primary source.
- Medium: the source is credible, but evidence is limited or not cross-checked.
- Low: single-source, vague, promotional, or weakly supported.

### 4. Produce a decision brief

The final output should support a decision, not just store information.

## Output format

```text
# Research conclusion: {topic}

## One-line answer
Answer the research question in one sentence.

## Key judgments
1. Judgment: ...
   Confidence: high / medium / low
   Evidence: source A, source B
   Meaning: what this implies for the current question

2. Judgment: ...
   Confidence: ...
   Evidence: ...
   Meaning: ...

## Disagreement and uncertainty
- Which sources disagree
- Which conclusions lack evidence
- What needs follow-up

## Recommended actions
- Do now: ...
- Avoid for now: ...
- Validate next: ...

## Sources
- <a href="..." target="_blank" rel="noreferrer">Source name</a>
```

## Quality bar

A good research synthesis must:

- Trace every key judgment back to sources.
- Separate facts, interpretation, and speculation.
- Avoid treating one media story as an industry conclusion.
- Include next actions, not only a summary.
- Be easy to challenge, verify, and update.

## Example

```text
# Research conclusion: Should AI coding tools enter enterprise development workflows?

## One-line answer
Yes, but start with low-risk generation, tests, and documentation tasks before allowing changes to production repositories.

## Key judgments
1. Judgment: Enterprise AI coding tools are adding permissions, audit trails, and organization-level policy.
   Confidence: high
   Evidence: GitHub Changelog, vendor documentation
   Meaning: Pilot conditions are better than before, but internal governance still matters.

## Disagreement and uncertainty
- Vendors differ in how they explain code data use, model training, and log retention.
- Long-term quality gains still need internal benchmarks.

## Recommended actions
- Do now: run a two-week pilot on one or two non-critical repositories.
- Avoid for now: do not allow agents to modify protected production branches directly.
- Validate next: track PR acceptance, human edit rate, and security findings.
```

## Practical advice

The value of this skill is compression, not length. Prefer verifiable judgments over background exposition.
