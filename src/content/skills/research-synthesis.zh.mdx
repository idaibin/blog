这个 Skill 适合在资料来源较多、观点不一致、需要做判断时使用。它不是简单总结，而是把来源拆成证据、结论、分歧和行动建议。

## 触发条件

当用户提出以下请求时触发：

- “帮我研究一下这个方向是否值得做。”
- “把这些链接整理成结论。”
- “这些资料互相矛盾，帮我判断。”
- “给我一份可以决策的研究摘要。”
- “不要只总结，要告诉我下一步怎么做。”

不适合触发的情况：

- 只有一个来源，普通摘要即可。
- 用户只要翻译或改写。
- 用户明确要求创意写作，而不是判断。

## 输入要求

执行前先确认至少有这些信息：

```text
研究问题：...
目标读者：...
来源材料：链接 / 摘要 / 原文片段
时间范围：...
输出用途：决策 / 文档 / 产品设计 / 技术选型 / 内容写作
约束：字数、语言、是否需要引用、是否需要保守判断
```

如果缺少研究问题或来源材料，先问清楚，不要直接开始。

## 执行流程

### 1. 拆来源

对每个来源提取：

- 来源名称
- 发布时间
- 作者或机构
- 核心主张
- 支撑证据
- 可能的利益相关或偏向
- 是否是一手来源

### 2. 聚类

把材料归入几类：

- 已确认事实
- 产品或技术变化
- 数据或实验结论
- 行业观点
- 推测和未验证判断
- 反对意见或风险

### 3. 校准可信度

给每个结论标注可信度：

- 高：多来源一致，且至少有一个一手来源。
- 中：来源可信，但证据有限或缺少复核。
- 低：只有单一来源、表述模糊或明显带宣传目的。

### 4. 输出决策摘要

最终输出不要堆资料，要服务决策。

## 输出格式

```text
# 研究结论：{主题}

## 一句话结论
用一句话回答研究问题。

## 关键判断
1. 判断：...
   可信度：高 / 中 / 低
   依据：来源 A、来源 B
   含义：这对当前问题意味着什么

2. 判断：...
   可信度：...
   依据：...
   含义：...

## 分歧和不确定性
- 哪些来源说法不一致
- 哪些结论缺少证据
- 哪些事情需要继续观察

## 建议动作
- 现在可以做：...
- 暂时不要做：...
- 下一步验证：...

## 来源清单
- <a href="..." target="_blank" rel="noreferrer">来源名称</a>
```

## 质量标准

一份合格的研究归纳必须满足：

- 每个关键判断都能追溯到来源。
- 明确区分事实、判断和推测。
- 不把单个媒体报道当成行业结论。
- 有下一步动作，而不是只给摘要。
- 结论可以被质疑、复查和更新。

## 示例

```text
# 研究结论：AI 编程工具是否适合进入企业研发流程

## 一句话结论
可以试点，但应先限制在低风险代码生成、测试补全和文档任务，不宜直接开放到核心生产仓库。

## 关键判断
1. 判断：企业级 AI 编程工具正在补齐权限、审计和组织策略。
   可信度：高
   依据：GitHub Changelog、厂商官方文档
   含义：试点条件比早期更成熟，但仍要看企业自身治理能力。

## 分歧和不确定性
- 不同厂商对代码数据使用、模型训练和日志保留的说明不一致。
- 长期质量收益仍需要内部基准测试验证。

## 建议动作
- 现在可以做：选 1-2 个非核心仓库做两周试点。
- 暂时不要做：不要直接让 agent 修改生产分支。
- 下一步验证：记录 PR 通过率、人工修改比例和安全问题数量。
```

## 使用建议

这个 Skill 的价值在于收敛信息，不在于写得长。优先输出可验证判断，少写背景铺垫。
