做 Rustzen、Feeds Hub 和 blog 时，我发现自己在不同会话里反复说同一批话：先读仓库规则，不要直接改 `main`，确认修改范围，保护未提交内容，最后运行验证并说明风险。

这些要求并不复杂，但漏掉任何一条，都可能把一次正常修改变成额外返工。于是我开始把它们从聊天记录里移出来，放进 `AICraft`。

项目地址：[`idaibin/aicraft`](https://github.com/idaibin/aicraft)

## Prompt 描述任务，Skill 固化执行契约

我现在用一个很简单的标准区分它们。

Prompt 描述要完成的任务。例如：检查某个 API 契约，改一篇文章，或者为一个功能列出实现步骤。它可以独立执行，也可以持续修改和复用；是否保留，取决于这类任务会不会再次出现。

Skill 描述这类任务以后都怎么做。它需要稳定的触发条件、执行顺序、禁止事项、验证方式和输出格式。只要任务反复出现，并且错误往往来自步骤缺失，它就更适合被写成 Skill。

AICraft 目前把资产分成几层：

- `prompts/` 保存可独立使用、持续演进的任务指令；
- `skills/` 保存可安装、可复用的执行流程；
- `docs/standards/` 保存跨任务都要遵守的分支、自动化和内容质量规则；
- `docs/templates/` 保存任务文档骨架；
- `docs/task-registry.md` 记录下游仓库里的具体自动化任务。

这个拆分不是为了把所有提示词都工程化，而是让真正稳定的规则有明确归属。

## 一个 Skill 至少要回答什么

只有一段长 Prompt，还不能算稳定的 Skill。对我来说，它至少要回答：

1. 什么请求应该触发它，什么请求不应该；
2. 它拥有哪部分工作，和相邻 Skill 如何分工；
3. 执行顺序是什么；
4. 哪些事实、文件、分支或命令不能被擅自改变；
5. 完成后如何验证；
6. 结果应该以什么形式返回。

这些内容写清楚后，Skill 才能被审查。否则它只是另一段更长、更难发现错误的提示词。

## 为什么要放进 Git

把规则放进仓库的直接好处不是“AI 变聪明了”，而是规则终于可以像代码一样被比较和回退。

一次修改影响了触发范围，可以检查 eval cases；输出格式变了，要同步 Agent 元数据；新增引用文件，要确认主 Skill 是否真的链接到它。质量问题不再只靠读者感觉，而是能落到具体规则和回归样例上。

当然，这也有成本。规则会过时，过度细化会让 Skill 只适合一个仓库，写得太抽象又失去约束力。AICraft 需要持续删掉已经失效的内容，而不是只往里面加说明。

## 仓库边界仍然要保留

```text
aicraft = 可复用能力和共享标准
blog = 长文发布与经验记录
feeds-hub = 信息流自动化实现
rustzen = 产品和工程代码
```

AICraft 不保存生成后的博客文章，也不保存具体 Feed，更不保存 Rustzen 的产品代码。通用规则可以被多个仓库消费，但每个仓库仍然要保留自己的运行命令、目录边界和业务事实。

现在安装这些 Skills 仍然使用标准的 skills.sh 流程：

```bash
npx skills add https://github.com/idaibin/aicraft
npx skills update
```

我把 AICraft 当成 AI 协作的配置仓库，而不是提示词收藏夹。它没有消除模型的不确定性，只是把一部分原本依赖临时提醒的工作，变成了可版本化、可验证的项目资产。
