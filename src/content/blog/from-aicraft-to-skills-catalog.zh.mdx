> 更新（2026-07-21）：补充当前 15 个 Skills 的职责边界，以及 UI 设计、领域建模、Review、Delivery 和外部研究的最新归属。

最初建立 AICraft，是因为我在不同项目和对话里反复强调同一批要求：先读仓库规则、确认修改范围、保护未提交内容、不要直接改 `main`，最后运行验证并说明风险。

把这些要求写进 Git 是有效的。问题出在后面：AICraft 同时开始保存 Prompt、共享标准、自动化模板、评测基础设施和越来越多的 Skills。它逐渐从“可以复用的能力集合”变成了一个需要理解内部路由和共享规则的框架。

这和 Agent Skills 的安装方式并不匹配。用户安装一个 Skill 时，真正需要的是一个边界清楚、可以单独运行的能力包，而不是先理解整个仓库。

## 数量增加不等于能力更完整

当时有些内容看起来适合单独做成 Skill，例如 Planning 和 Diagnosis。继续使用后，我发现它们更像默认工作纪律。

复杂任务本来就需要拆分范围、依赖和验收条件；具体故障本来就应该先复现、收集证据、验证假设，再确认根因。这些规则不依赖特定技术栈，也不需要独立脚本或参考包。把它们做成 Skill，只会增加一次额外的触发判断。

类似问题也出现在 Prompt 上。通用开发、生成 `AGENTS.md`、Review 和 Commit 等 Prompt，与 Codex 默认能力、个人配置和仓库规则高度重叠。它们继续存在，不会增加新的能力，只会产生多个互相接近的入口。

我最后采用了更严格的判断：一个内容只有在拥有独立触发条件、授权边界、资料、流程或专项验证时，才值得成为 Skill。

## 重构后的归属

现在的结构更简单：

```text
~/.codex/AGENTS.md = 跨仓库稳定成立的个人工作偏好
仓库 AGENTS.md     = 当前项目的结构、命令和交付规则
blog / Prompts     = 可以阅读、复制和修改的 Prompt 与配置
idaibin/skills     = 可以独立安装的专业能力包
blog / Notes       = 设计判断、实践过程和历史经验
```

Prompt 不再放在 Skills 仓库里。它们由 blog 作为公开内容维护，目前只保留少量职责清楚、仍然需要复制使用的版本。

Skills 仓库则收敛为 [`idaibin/skills`](https://github.com/idaibin/skills)。每个包只能依赖自己的 `SKILL.md`、`references/`、`assets/` 和 `scripts/`；仓库根目录的文档、合同和评测脚本只服务维护，不作为安装后的隐藏依赖。

## 我保留了什么

Prompt 最终只保留四类：

- Codex 工作配置；
- 前后端 API 契约审查；
- 对话记录与经验复盘；
- 可配置的定时主题摘要。

Skills 保留的是仓库理解、领域建模、前端和 Rust 实现、专项审计、Review、Delivery、浏览器与客户端验证等明确能力。它们可以组合，但没有强制顺序。

当前目录保留 15 个公开能力包，但边界比数量更重要。`ui-design` 默认只为一个页面或流程提供 Feature UI 方案；只有共享 token、组件语义、variant 或整体视觉语言发生变化时，才进入 Design System profile。`domain-modeling` 默认处理跨功能共享的业务术语、规则和边界场景，生命周期与 bounded context 只在复杂度确实需要时启用。`product-spec` 描述用户可见行为和验收，不拥有 API；需要长期导航时，由 `repo-map` 记录真实 authority、消费者和验证入口。

Review 与交付也被拆开。`repo-review` 只读当前 Worktree/index、固定 SHA/range 或已校验的 review package；`repo-delivery` 默认把已审查变更按语义意图分类提交，只有明确要求或完整范围确实只有一个不可分割意图时才使用单提交。整合其他分支时，它根据提交边界和仓库策略决定保留有意义的 commit，还是 squash 掉嘈杂、修补型或单一结果的历史；commit 授权不会自动推导出 push。当 Codex 确实需要独立外部意见时，`ask-chatgpt` 负责向 ChatGPT 发起经过明确授权的审查、研究、视觉探索或决策挑战，返回内容仍要回到对应 basis 本地核实。

一些内容则只保留在 Git 历史里。一次性赛事海报、依赖本机路径的产品图标 Prompt、已经被其他能力覆盖的通用工作流，都不再作为公开入口。历史存在过，不代表它们需要永远占据当前目录。

## 删除也是维护

以前我更关注怎样把反复出现的要求沉淀下来。现在我更关心另一件事：这条规则应该由谁负责，以及它是否还需要独立存在。

如果同一个行为同时写在个人配置、Prompt、Skill 和仓库文档里，更新任何一处都可能产生漂移。把内容移动到正确的层级，通常比增加新的规则更有效。

现在安装 Skills 使用标准的 skills.sh 流程：

```bash
npx skills@latest add idaibin/skills
```

这次重构没有否定 AICraft 的价值。相反，它把已经验证过的内容从一个过宽的仓库中拆出来：专业能力进入 Skills，公开 Prompt 和经验进入 blog，稳定偏好进入个人配置。真正需要长期维护的资产因此变少了，也更容易说明白。
