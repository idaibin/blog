# Rustzen Architecture v1.0

## 1. Overview

Rustzen is a CLI-driven, SQLite-first, multi-repository Rust ecosystem designed around strict modular boundaries and deterministic execution behavior.

This document defines the final unified architecture standard derived from full rustzen-* repository audit.

---

## 2. Core Principles

- Local-first (SQLite as default persistence layer)
- Explicit runtime control (no hidden framework behavior)
- Strict layered architecture
- CLI-first operational model
- Observable system behavior (mandatory tracing)

---

## 3. System Architecture Model

```
CLI / GUI / API
        |
     Service
        |
  ----------------
  |              |
 Core           DB
  |              |
Domain       SQLite
```

---

## 4. Repository Structure Standard

Every rustzen-* repository MUST follow:

```
apps/
  cli/       -> command entry only
  server/    -> API entry only
  gui/       -> UI layer (Tauri)

crates/
  core/      -> pure business logic
  service/   -> orchestration layer
  db/        -> persistence layer only
  domain/    -> types + invariants
  utils/     -> shared utilities

infra/
  config/
  logging/
  runtime/
```

---

## 5. Layer Rules

### Handler Layer (apps)
- input parsing only
- validation only
- forwards to service
- NO DB access

### Service Layer
- orchestration only
- composes core + db

### Core Layer
- pure deterministic logic
- no IO

### DB Layer
- persistence only
- SQL + migrations only

---

## 6. Dependency Rules

Strict direction:

```
Handler -> Service -> Core -> DB
```

### Forbidden:
- handler → db
- core → db
- db → service

---

## 7. CLI Standard (rz System)

```
rz <domain> <action>
```

Examples:

```
rz clear scan
rz clear analyze
rz clipboard list
rz analytics report
```

Rules:
- CLI contains no business logic
- CLI maps 1:1 to service layer

---

## 8. God Module Definition

A module is considered a GOD MODULE if:

- mixes 2+ responsibilities (service + db + logic)
- exceeds ~800 LOC
- owns lifecycle + persistence + orchestration

---

## 9. System-Wide Findings Summary

Common issues across rustzen-* repositories:

- Service layers too broad (god services)
- Storage layers mixing persistence and logic
- CLI bypassing service boundaries
- Inconsistent module ownership patterns

---

## 10. Risk Heatmap

| Repo | Risk | Issue |
|------|------|------|
| rustzen-zipper | HIGH | CLI monolith |
| rustzen-clear | HIGH | ZenService overloading |
| rustzen-clipboard | HIGH | storage/capture coupling |
| rustzen-analytics | MEDIUM | report service overgrowth |
| rustzen-inspect | MEDIUM | scheduler coupling |
| rustzen-admin | MEDIUM | infra/db overreach |

---

## 11. Refactor Strategy (Global)

- split god modules into service + core + db
- enforce strict dependency direction
- standardize CLI mapping across all repos
- isolate persistence logic into db layer

---

## 12. Final Target State

Rustzen becomes:

> A strictly layered, CLI-driven, SQLite-first Rust ecosystem with enforceable architecture boundaries, deterministic module decomposition, and consistent cross-repo design.
