# Final Architecture Consistency: rustzen-inspect

## Scope

Repository reviewed in this pass: `rustzen/rustzen-inspect`.

This is a strict incremental consistency pass for the existing `rustzen-inspect` audit. It does not change the source repository, does not create a PR, and does not introduce a new system or abstraction.

Reviewed files in this pass:

- `Cargo.toml`
- `README.md`
- `docs/architecture.md`
- `docs/guides/backend.md`
- `zen-server/src/main.rs`

## PASS 1 — Incremental File-Level Findings

### `Cargo.toml`

Observed:

- The root remains a Rust workspace with four backend members: `zen-core`, `zen-common`, `zen-server`, and `zen-agent`.
- The workspace dependency graph keeps `sqlx` SQLite-only through `default-features = false` and explicit `sqlite`, `migrate`, and `runtime-tokio-rustls` features.
- Shared dependencies include Axum, Tokio, cron scheduling, Figment YAML config, logging, zip packaging, and common utility crates.

Consistency result:

- PASS at workspace boundary level.
- This confirms `rustzen-inspect` is a multi-process Rust workspace, not a single-server or single-CLI repository.
- The root dependency contract remains compatible with the Rustzen Architecture v1 SQLite-first rule.

### `README.md`

Observed:

- The product is documented as a self-hosted monitoring platform with `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common` as separate parts.
- The permission model is route-metadata based, with `superadmin`, `admin`, and `viewer` permissions described explicitly.
- Deployment is concrete: production root is `/opt/rustzen-inspect`, runtime files live under `bin/`, `config/`, `data/`, `logs/`, and `versions/`, and frontend output is served from `data/web/dist/`.
- Server and agent both use SQLite for runtime data.

Consistency result:

- PASS at product and deployment description level.
- The README exposes the true process split and runtime shape; it does not present the repository as a generic admin template.

### `docs/architecture.md`

Observed:

- The repository is explicitly classified as a protected Web/Rust multi-process exception.
- The document states that `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common` must not be collapsed into `apps/server` + `apps/web` as part of general standardization.
- Boundaries are explicit: server features stay under `zen-server/src/features/<feature>/`, agent business code stays under `zen-agent/src/features/<feature>/` or `zen-agent/src/task/`, and shared feature-neutral code stays in `zen-core` or `zen-common`.
- Runtime facts remain fixed around SQLite, YAML config, `PathResolver`, legacy runtime migration, UTC timestamps across server-agent boundaries, and preserved production directories.

Consistency result:

- PASS at repository-boundary level.
- This is a valid Rustzen Architecture v1 protected-layout exception.
- The remaining audit risk must be scored inside process entrypoints, schedulers, task catalogs, and domain services rather than root layout.

### `docs/guides/backend.md`

Observed:

- Backend layering is explicit: `mod.rs` wires modules/routes, `handler.rs` handles HTTP, `service.rs` handles orchestration/validation/transactions, `repo.rs` handles persistence, and `types.rs` owns local DTOs/rows.
- Server rules prohibit services bypassing repos and repos crossing feature boundaries.
- Monitoring-specific rules keep V1 TCP probing in `zen-server` and require Tokio TCP probing rather than shelling out.
- Agent rules require long-running collection, aggregation, cleanup, and scheduling under `zen-agent/src/task/`, while local collection, local persistence, and server reporting must stay separate steps.
- SQLite remains the only runtime database backend.

Consistency result:

- PASS as a written backend guide.
- The guide is strong enough to evaluate ordinary request/response features.
- The guide also makes the final risk visible: several actual files comply with the broad location rule, but contain high internal role density.

### `zen-server/src/main.rs`

Observed:

- Server startup owns config loading, path resolution, logger setup, runtime directory creation, legacy DB/web migration, SQLite initialization, `AppState` construction, CORS cache loading, current-project initialization, system task bootstrap, monitor scheduler startup, permission cache cleanup, router creation, permission sync, and HTTP serving.
- System task bootstrap receives database, report, alert, node, and agent services, plus retention settings.
- Monitor scheduling starts directly from the entrypoint after system task bootstrap.
- Permission sync happens after route creation through registered permission codes.

Consistency result:

- PARTIAL at entrypoint-thinness level.
- The file is readable and explicit, but it is broader than a thin startup dispatcher.
- The risk is runtime lifecycle concentration, not domain logic leakage.

### `zen-server/src/infra/app.rs`

Observed:

- `AppState` stores server start time, server address, database pool, and an `Arc<Services>` registry.
- `create_router` merges auth, node, agent, bootstrap, home, dashboard, deploy, alert, monitor, reports, CORS, project, system user/role/menu/task/dict/log/status routers, static avatar resources, auth middleware, operation-log middleware, CORS middleware, upload limits, compression, and SPA fallback serving.
- `Services` holds a broad registry: auth, node, bootstrap, deploy, alert, monitor, server-period reports, agent, cors, project, user, role, menu, task, dict, status, and operation log.
- `Services::new` constructs almost every feature repository and service in one function.

Consistency result:

- PARTIAL at composition-root level.
- The file is still mostly assembly code, so this is not a feature-boundary failure.
- It remains a central pressure point because nearly every server feature dependency is visible in one module.

### `zen-server/src/features/system/task/service.rs`

Observed:

- `SystemTaskService` owns task persistence, timezone, task catalog state, and a runtime `JobScheduler`.
- `bootstrap_tasks` constructs the catalog, registers cron jobs, reads next ticks, syncs task metadata, starts the scheduler, and stores catalog/scheduler state.
- `start_task_by_key` resolves catalog entries and dispatches manual or scheduled runs.
- `start_task_run` records running/skipped rows, enforces a per-task run lock, spawns async execution, and writes terminal status back to persistence.

Consistency result:

- PARTIAL at service-boundary level.
- This is a coherent system-task boundary, but it combines scheduler lifecycle, catalog access, run state machine, persistence, conflict handling, and async execution management.
- It confirms that scheduler ownership must be audited as a separate axis in multi-process Rustzen products.

### `zen-server/src/features/system/task/catalog.rs`

Observed:

- The catalog defines service dependencies, retention config, scheduled task shape, task specs, task kinds, executor mapping, cleanup targets, cleanup executors, report executor, alert sync executor, and tests.
- `TASK_SPECS` is deterministic and encodes six cleanup tasks, one report scheduler, and one alert rule sync task.
- Cleanup target execution calls cross-feature cleanup methods on the infra `Database`.
- The test protects stable task keys and schedules.

Consistency result:

- PARTIAL at catalog-boundary level.
- The task schedule contract is deterministic and test-backed.
- The catalog mixes declarative task metadata, executor construction, concrete cleanup/report/alert behavior, and retention dispatch.
- This is a centralization choice, not a repository-layout failure.

### `zen-server/src/features/reports/server_period/service.rs`

Observed:

- `ServerPeriodService` owns report listing, detail retrieval, status lookup, source-data detection, generation, metric ingestion projection, alert event ingestion, rollup construction, payload construction, and data quality coordination.
- `ingest_node_metrics` projects node metrics into report metrics through transaction-aware definition/sample upserts.
- `upsert_metric_projection_tx` defines metric definitions, pushes gauge/counter/rate samples, computes time offset, computes network totals/rates, reads prior counter samples, and writes definitions and samples.
- Backfill reads historical node metrics and replays projection through the same service method.

Consistency result:

- MEDIUM-HIGH localized domain-service density.
- The file belongs to the correct `reports/server_period` feature, but it is broader than ordinary service orchestration.
- The service combines ingestion projection, report generation, rollup behavior, payload/data-quality construction, and alert/report coordination.

### `zen-agent/src/main.rs`

Observed:

- Agent startup owns deployment preflight, config loading, path resolution, logger setup, runtime directory creation, legacy DB migration, SQLite initialization, AppState construction, HTTP client construction, one-time node summary reporting, long-running collection task startup, alert config sync startup, router creation, and HTTP serving.
- `start_collection_task` constructs a collector, interval timer, alert evaluation channel, alert engine worker, aggregation scheduler, and nested async collection loop.
- The collection loop performs local insert, aggregation, server reporting, and alert dispatch in sequence.
- HTTP response handling and agent-to-server metric reporting helpers also live in the entrypoint file.

Consistency result:

- FAIL at entrypoint-thinness level, while behavior remains aligned with the guide.
- The runtime steps are explicit and ordered, but the entrypoint contains long-running collection/reporting orchestration instead of only startup dispatch.
- This is the strongest agent-side consistency issue.

### `zen-agent/src/task/scheduler.rs`

Observed:

- `Scheduler` owns agent config, `Aggregator`, `Cleaner`, SQLite pool, cleanup timezone, and last-cleanup-day state.
- `run_aggregation_once` triggers minutely, hourly, daily, monthly aggregation and cleanup in one call.
- The scheduler performs direct SQL existence checks against `metrics_aggregated` before dispatching some aggregation paths.
- Cleanup timing is timezone-aware and test-backed.

Consistency result:

- PARTIAL at task-scheduler level.
- The scheduler is operationally deterministic, but combines scheduling, aggregation dispatch, cleanup dispatch, cleanup state, and direct SQL checks.
- The direct SQL check is a small persistence-boundary leak inside task orchestration.

### `zen-agent/src/task/aggregator.rs`

Observed:

- `Aggregator` owns SQLite access, active network filtering, raw metric aggregation, aggregate-to-aggregate rollup, JSON decoding, CPU/load/memory/swap/disk/network/process/uptime math, and upsert coordination.
- `aggregate_minutely` reads raw rows from `node_status_raw`, computes aggregated metrics, and upserts the result.
- `aggregate_from_existing` reads existing aggregates, calculates higher-level rollups, and upserts the result.
- Calculation paths include CPU core JSON parsing, disk aggregation, network aggregation, process statistics, uptime selection, and interval-specific metric behavior.

Consistency result:

- FAIL at task-module density level.
- The module is correctly located under `zen-agent/src/task/`, but combines persistence, JSON decoding, filtering, metric math, rollup construction, and write coordination.
- This remains the highest-density agent computation surface.

## PASS 2 — Per-Repo Consistency Checks

### Consistent

- The protected multi-process root layout is documented and implemented.
- SQLite-first remains consistent across workspace dependencies, docs, and runtime statements.
- Server/agent shared crates are documented as shared primitives/utilities, not feature business containers.
- The runtime directory contract is concrete and stable.
- Monitoring, report, system-task, and agent task locations mostly match their documented feature roots.

### Pressured

- Server `main.rs` is broad but still mostly lifecycle assembly.
- Server `infra/app.rs` is a broad composition root for nearly all features.
- System task service combines scheduler lifecycle, run recording, locking, async dispatch, and terminal persistence update.
- System task catalog combines task specs, executor construction, cleanup target dispatch, report scheduling, and alert rule sync.
- Server period report service is a high-density reporting/ingestion/projection surface.
- Agent `main.rs` combines startup with long-running collection, aggregation, reporting, and alert dispatch.
- Agent scheduler owns both orchestration and direct SQL existence checks.
- Agent aggregator owns persistence, JSON decoding, metric math, filtering, rollup construction, and upsert behavior.

### Final consistency delta

The previous `rustzen-inspect` audit classified this repository as **MEDIUM risk**. This final pass confirms that rating.

The risk is not caused by repository layout. The repository layout is one of the clearest in the Rustzen set. The risk is caused by internal execution density across server system tasks, server report generation, agent entrypoint orchestration, agent scheduler logic, and metric aggregation.

## PASS 3 — Unified Rustzen Architecture Rules Update

### Final risk rating

`rustzen-inspect` remains **MEDIUM risk**.

Reason:

- Repository layout: PASS.
- Documentation boundary: PASS.
- SQLite-first runtime model: PASS.
- Server composition density: PARTIAL.
- Server report/task service density: PARTIAL to MEDIUM-HIGH localized risk.
- Agent entrypoint and aggregation density: FAIL locally, but contained inside a documented multi-process product.

The final rating stays MEDIUM because the high-density areas are contained inside correct process and feature roots. They are not spread across unrelated repositories or hidden in shared crates.

### Rustzen Architecture v1 update

Add this audit rule:

> Multi-process Rustzen repositories must be scored on two axes: protected process-root validity and internal execution-density risk. A repository may pass protected layout review while still carrying MEDIUM risk when entrypoints, schedulers, task catalogs, report services, and aggregation modules combine multiple runtime roles.

Add this consistency rule:

> In Rustzen Architecture v1, `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common` may remain protected roots only when the deployment model and documentation make those process boundaries explicit. That exception does not waive entrypoint-thinness, scheduler ownership, service density, or persistence-boundary review.

Add this scheduler rule:

> Schedulers must be audited by ownership model: persisted system scheduler, in-memory monitor/probe scheduler, local agent aggregation scheduler, and report-generation scheduler are separate architecture surfaces even when all are valid inside the same product.

Add this aggregation rule:

> Metric aggregation modules may be computation-heavy, but audit classification must distinguish SQL access, JSON decoding, active network filtering, metric math, rollup construction, cleanup coordination, and upsert persistence as separate responsibility groups.

### Repo-specific final check

- Do not classify `rustzen-inspect` as migration debt because it keeps `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common`.
- Do not score it as a generic admin template.
- Do not apply the single-process `apps/server + apps/web` rule to this repository.
- Score it as a local-first multi-process monitoring product with bounded roots and medium internal service/task density.

## Blog Series Article

### Designing a Local-First Monitoring System Without Collapsing Process Boundaries

A local-first monitoring product is not naturally a single process. It has a server that manages nodes, policies, reports, permissions, and deployment. It has an agent that collects metrics, stores local samples, evaluates alerts, reports to the server, and exposes a small node-side API. It has a web interface. It also needs shared primitives for auth, paths, logging, deployment validation, and common models.

`rustzen-inspect` is useful because it makes that shape explicit. The repository keeps `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common` as protected roots. That is not accidental legacy structure. It is part of the product contract. The deployment model also depends on it: production binaries, systemd units, SQLite files, config files, logs, versions, and frontend static output all have stable runtime locations.

That is the first architecture lesson: standardization does not always mean forcing every repository into the same directory shape. For Rustzen Architecture v1, a protected layout is valid when the runtime model requires separate process roots and the documentation says so clearly.

The second lesson is stricter. A valid repository layout does not automatically make the internal architecture healthy. In `rustzen-inspect`, the pressure appears inside execution modules. The server system task service owns scheduler lifecycle, run locks, run persistence, async execution, and terminal status updates. The task catalog owns both task metadata and executor construction. The server period report service owns ingestion projection, report generation, data quality, rollups, alert projection, and payload construction.

The agent side has the same pattern. `zen-agent/src/main.rs` does more than startup: it starts long-running collection, stores samples locally, runs aggregation, reports to the server, and dispatches alerts. The agent scheduler coordinates minutely, hourly, daily, and monthly aggregation plus cleanup. The aggregator owns SQL access, JSON decoding, network filtering, metric math, rollup construction, and upsert behavior.

None of this means the repository boundary is wrong. It means monitoring software naturally accumulates runtime responsibility. The audit model must therefore separate two questions: is the process layout valid, and are the execution modules still auditable?

For a Rustzen-style local-first product, the goal is not to make every repository look identical. The goal is to keep the runtime shape honest. A monitoring product may remain multi-process, SQLite-first, and deployment-simple. But its schedulers, aggregators, report generators, and entrypoints must be treated as first-class architecture surfaces.

`rustzen-inspect` confirms the rule: process boundaries can be protected, but execution density cannot be ignored.
