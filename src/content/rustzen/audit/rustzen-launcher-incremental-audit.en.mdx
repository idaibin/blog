## PASS 1 — File-Level Inspection

### `README.md`

Observed:

- The repository is explicitly classified as a Rustzen Client/Tauri desktop prototype.
- It states that the repo is not a release baseline and not a reusable Rustzen desktop template.
- Runtime stack is React 19, TypeScript, Vite, TanStack Router/Query, and Tauri 2.
- Release and updater states are both explicitly `none`.
- Root `src-tauri` is documented as a prototype-only layout exception.
- `pnpm check` is documented as write-mode because it runs `biome check --write`.

Architecture classification:

- PASS at documentation boundary level.
- The README prevents this prototype layout from being treated as a product baseline.
- The repository has no current claim of publishable release readiness.

### `docs/prototype-graduation.md`

Observed:

- The document classifies the repository as a Client/Tauri desktop prototype.
- It protects the normal Rustzen desktop layout as `zen-core` plus `zen-gui/src-tauri`.
- It explicitly states that the current root `src-tauri` layout must not be copied into new desktop projects.
- Release, updater, signing, notarization, publishing, and GitHub release workflows are out of scope until graduation is approved.
- Code readiness blockers are recorded: mock search data, placeholder login route, non-minimized permissions, and undocumented clipboard polling ownership.
- The graduation checklist requires identity review, real local data or command contract, capability minimization, updater decision, bundling approval, and verified commands.

Architecture classification:

- PASS as governance documentation.
- The doc creates a strong exception boundary.
- The main risk is not missing documentation; it is that runtime code already has background behavior before product ownership rules exist.

### `package.json`

Observed:

- Package is private and versioned as `0.1.0`.
- Scripts expose Vite dev/build, Tauri dev, Biome lint/format/check, and preview.
- `postbuild` runs `rz-zip` through `@rustzen/zipper`.
- `check` runs `biome check --write`.
- Vite is overridden to `rolldown-vite@latest`.

Architecture classification:

- PASS for prototype tooling.
- Distribution is intentionally local/private.
- `postbuild` creates a cross-repo dependency on `@rustzen/zipper`, but it is contained in frontend packaging and does not leak into Tauri Rust runtime.

### `src-tauri/Cargo.toml`

Observed:

- Single Tauri crate named `rustzen-launcher`, version `0.1.0`, edition `2021`.
- Library crate is named `rustzen_launcher_lib` and exposes `staticlib`, `cdylib`, and `rlib` crate types.
- Dependencies include Tauri 2, opener, clipboard manager, log, Tokio with `full`, serde, and serde_json.
- macOS private API is enabled through Tauri features.
- Autostart and global shortcut plugins are enabled for non-mobile targets.

Architecture classification:

- LOW-MEDIUM risk.
- Dependency surface is acceptable for a launcher prototype.
- `tokio = full` is broader than the current Rust source needs, but the risk is bounded because this is not a release baseline.

### `src-tauri/src/main.rs`

Observed:

- The binary entrypoint only delegates to `rustzen_launcher_lib::run()`.
- Windows console suppression is scoped to non-debug builds.

Architecture classification:

- PASS.
- The binary entrypoint is thin and has no business logic.

### `src-tauri/src/lib.rs`

Observed:

- The file creates a Tokio runtime manually before building the Tauri app.
- Runtime entry is held with `_guard = runtime.enter()`.
- The Tauri builder installs log, autostart, clipboard manager, global shortcut, and opener plugins.
- `setup` clones the app handle and spawns `start_clipboard_monitor` as a background task.
- Tauri run errors are logged and not propagated.

Architecture classification:

- MEDIUM localized risk.
- This file is still compact, but it owns runtime creation, plugin registration, background task startup, and application lifecycle wiring.
- The background task has no explicit shutdown contract at the app lifecycle boundary.

### `src-tauri/src/utils/mod.rs`

Observed:

- The module only exposes `clipboard`.

Architecture classification:

- PASS.
- No boundary concern in the module file itself.

### `src-tauri/src/utils/clipboard.rs`

Observed:

- `start_clipboard_monitor` creates an `AtomicBool` initialized to `true`.
- No external owner receives the flag, so the loop has no observable stop path.
- Clipboard text is polled every 500 ms.
- Last text content is retained in memory to suppress duplicate events.
- On content change, the code emits `clipboard://change` with an empty payload.
- Clipboard read failures are logged as warnings on every poll failure.

Architecture classification:

- MEDIUM risk within a prototype.
- The function mixes polling lifecycle, clipboard adapter access, duplicate suppression, event emission, and logging policy.
- Event contract is not yet meaningful because no payload, ownership, retry/backoff, or shutdown semantics are defined.

### `src-tauri/tauri.conf.json`

Observed:

- Product name and window title are `rustzen-launcher`.
- Identifier is `com.daibin.rustzen-launcher`.
- `macOSPrivateApi` is enabled.
- Main window is transparent, undecorated, fixed-size, skip-taskbar, and centered.
- Security CSP is `null`.
- Bundling is disabled.

Architecture classification:

- PASS as prototype configuration.
- Release-sensitive identity, CSP, private API, and bundle settings are correctly blocked by the prototype graduation document.
- These settings must remain classified as prototype state, not product default.

### `src-tauri/capabilities/default.json`

Observed:

- The default capability applies only to the `main` window.
- Permissions include core defaults and specific window permissions for show, hide, focus, drag, and size.

Architecture classification:

- PASS for the current UI behavior.
- Capability scope is small, but the graduation document is correct that release policy is not yet complete.

### `src/routes/index.tsx`

Observed:

- Home route owns the search UI, mock search data, keyboard navigation, selected index, and dynamic window resizing.
- `useGlobalShortcut` and `useClipboard` are both mounted in the route component.
- Mock search items are embedded directly in the route file.
- `handleItemSelect` currently clears UI state but does not execute any launcher command.

Architecture classification:

- MEDIUM prototype risk.
- The UI route is both interaction shell and mock command model.
- There is no real command registry, local data contract, or service boundary yet.

### `src/hooks/useClipboard.ts`

Observed:

- The hook subscribes to `clipboard://change` only in Tauri runtime.
- It guards duplicate registration with a ref.
- Cleanup handles cancelled registration and unlisten callback.
- The event handler only logs `Clipboard changed`.

Architecture classification:

- PASS as a frontend event subscriber.
- The hook confirms that the current clipboard event is only a notification pulse, not a product data contract.

### `src/hooks/useGlobalShortcut.ts`

Observed:

- The hook registers `CommandOrControl+Shift+R`.
- It unregisters the same hotkey before registering.
- On pressed state it toggles the window.
- It also attaches an Escape key listener to hide the window.
- Cleanup unregisters the shortcut and removes the key listener.

Architecture classification:

- PASS for prototype behavior.
- Shortcut ownership is isolated in one hook.
- Hotkey identity is hardcoded and not yet governed by product settings or identity policy.

### `src/utils/index.ts`

Observed:

- Window helper functions directly use the current Tauri window.
- `toggleWindow`, `hideWindow`, and `resizeWindow` are active helpers.
- `maximizeWindow` exists but is unused by the launcher route.
- `unmaximizeWindow` is a no-op placeholder.

Architecture classification:

- LOW-MEDIUM risk.
- Window utility concentration is acceptable in prototype state.
- Placeholder functions should not be interpreted as final product surface.

## PASS 2 — Cross-Module Consistency

### Documentation vs repository layout

Consistent:

- README and graduation doc both classify root `src-tauri` as a prototype-only exception.
- The Rust code matches a compact Tauri prototype rather than a reusable Rustzen desktop baseline.

Finding:

- The repository is correctly fenced as a prototype and should not be scored as a failed product layout.

### Runtime behavior vs governance blockers

Consistent:

- The graduation doc already records the clipboard polling behavior as lacking runtime ownership, failure policy, and event contract.
- The Rust implementation confirms this: `start_clipboard_monitor` has a perpetual polling loop and emits an empty event.

Finding:

- Runtime behavior is functional but not yet governed enough for product graduation.

### UI behavior vs launcher product contract

Consistent:

- The graduation doc records mock search data as a blocker.
- The route file confirms mock data and no real launcher execution contract.

Finding:

- The frontend is a UI exploration shell, not yet a product launcher workflow.

### Capability and release boundary

Consistent:

- Bundling is disabled in Tauri config.
- Updater and release workflows are explicitly blocked.
- Capabilities are small but not yet release-reviewed.

Finding:

- Release risk is contained by documentation and disabled bundling.

## PASS 3 — Unified Architecture Synthesis

### Confirmed repo status

`rustzen-launcher` is classified as **LOW-MEDIUM risk**.

Reason:

- It has strong prototype governance and low Rust code volume.
- The binary entrypoint is clean.
- The main architectural pressure is background runtime ownership and mock UI/command boundaries, not repository-wide service or DB coupling.

### Standard refinement from this pass

For Rustzen Client/Tauri prototype repositories:

- Root `src-tauri` is allowed only when the repository is explicitly documented as a prototype exception.
- Prototype exception docs must state that the layout is not reusable as a Rustzen desktop template.
- Background loops must declare ownership, shutdown behavior, event contract, and failure policy before graduation.
- Empty frontend events are allowed for prototype signaling but not for product contracts.
- Tauri identity, CSP, private API, capability scope, bundle targets, signing, notarization, updater, and artifact naming are release-boundary items.
- Mock UI data must remain classified as prototype evidence until backed by a real local command/data contract.

## Audit Delta

This pass adds a prototype-specific Rustzen rule:

> A Client/Tauri repository can retain a non-standard root `src-tauri` layout only while it is explicitly fenced as a prototype and release-disabled. The first graduation boundary is not layout migration by default; it is proof of runtime ownership, event contracts, capability review, product identity, and real local command/data behavior.

No source repository was modified in this pass. Only this audit document was added under `rustzen/blog`.
