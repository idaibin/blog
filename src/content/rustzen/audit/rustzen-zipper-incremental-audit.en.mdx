# Incremental Audit: rustzen-zipper

## Scope

Repository reviewed in this pass: `rustzen/rustzen-zipper`.

Reviewed files:

- `Cargo.toml`
- `README.md`
- `package.json`
- `index.js`
- `scripts/install.js`
- `scripts/verify-package.js`
- `src/main.rs`

## PASS 1 — File-Level Inspection

### `Cargo.toml`

Observed:

- Single binary package: `rustzen-zipper`, version `0.2.0`, edition `2024`.
- Dependency set directly includes time formatting, CLI parsing, glob matching, hashing, directory traversal, ZIP read/write, and JSON config parsing.
- No workspace, internal crate, module tree, or explicit library boundary exists.

Architecture classification:

- Binary-only CLI crate.
- Good for small distribution footprint.
- Weak for Rustzen architecture boundaries because command parsing, config, filesystem traversal, archive IO, and reporting are not separated at crate or module level.

### `README.md`

Observed:

- Public package is `@rustzen/zipper`.
- Published command surface is `rz-zip` with documented `rz-zip unpack`.
- Configuration source priority is documented as CLI explicit args, explicit config path, workspace config files, then CLI defaults.
- Release contract documents platform assets: macOS arm64, macOS x64, Linux x64 GNU, and Windows x64 MSVC.
- README is operationally useful and includes install, config, CI, troubleshooting, timestamp format, include/exclude, output, checksum, prefix, and dry-run behavior.

Architecture classification:

- Documentation has stronger product and release clarity than the Rust source layout.
- The documented feature set already implies separate domains: command entry, config loading, pack workflow, unpack workflow, release install, and package verification.
- There is a consistency concern: README exposes `rz-zip unpack`, while the Rust command marks `unpack` as hidden.

### `package.json`

Observed:

- npm package exposes a single binary command: `rz-zip`.
- Version matches `Cargo.toml` at `0.2.0`.
- `files` includes `index.js`, `README.md`, `scripts/install.js`, and `scripts/verify-package.js`.
- Description says `CLI & JS library`, but package runtime behavior is a binary wrapper only.
- CI command runs Rust build/test/clippy and npm package verification.

Architecture classification:

- npm layer is a distribution adapter, not an application layer.
- Metadata wording overstates JS library behavior.
- Release/package verification is present and useful, but not complete for the documented release asset matrix.

### `index.js`

Observed:

- Resolves platform binary from package-local `bin/`.
- Fails fast if binary is missing.
- Delegates all user arguments to the Rust binary with inherited stdio.

Architecture classification:

- PASS.
- This is a thin JS launcher.
- No business logic leaks into the JS wrapper.

### `scripts/install.js`

Observed:

- Detects target platform and architecture.
- Supports macOS arm64, macOS x64, Linux x64 GNU, and Windows x64 MSVC.
- Downloads `rustzen-zipper-<target-triple>` from GitHub Release tag `v${package.version}`.
- Installs the downloaded asset into `bin/rustzen-zipper` or `bin/rustzen-zipper.exe`.

Architecture classification:

- Distribution responsibility is isolated from Rust code.
- Installer behavior matches README’s documented release asset naming.
- Linux arm64 is not supported by this script, which is consistent with the documented asset list.

### `scripts/verify-package.js`

Observed:

- Verifies `Cargo.toml` and `package.json` version equality.
- Verifies npm dry-run contents.
- Copies the debug Rust binary into `bin/` and verifies `node index.js --version` output.
- Does not validate remote GitHub Release assets.

Architecture classification:

- PARTIAL PASS.
- Local package contract is verified.
- Documented release matrix is not verified in the local package check.

### `src/main.rs`

Observed:

- Root imports include CLI parsing, glob matching, JSON parsing, hashing, filesystem IO, timing, directory walking, ZIP read, and ZIP write in one file.
- `Cli`, `Commands`, `PackOptions`, `UnpackOptions`, `CompressionMode`, `OverwriteMode`, `ZipStats`, `PackRuntimeOptions`, and `PackConfig` all live in `main.rs`.
- `main()` parses CLI and dispatches directly into `run_pack` or `run_unpack`.
- `run_unpack` handles source validation, output directory creation, archive opening, zip-entry path safety, directory creation, file creation, copy, and logging.
- `run_pack` handles runtime option resolution, source validation, glob parsing, output directory creation, timestamp naming, overwrite behavior, compression options, archive prefix calculation, filesystem walking, include/exclude filtering, stats mutation, dry-run summary, zip writing, checksum output, and summary printing.
- Config loading, JSON parsing, config merging, compression parsing, overwrite parsing, boolean parsing, glob parsing, prefix stripping, checksum writing, inclusion matching, and summary printing are helper functions in the same file.

Architecture classification:

- FAIL for Rust layering.
- `src/main.rs` is a confirmed CLI monolith.
- The monolith is functional and compact, but it violates Rustzen Architecture v1 command boundary expectations.

## PASS 2 — Cross-File Consistency

### CLI surface

Finding:

- README documents `rz-zip unpack` as current user-facing behavior.
- Rust command definition hides `unpack` from generated help.
- This is a documentation/help discoverability mismatch, not evidence that the command is absent.

### npm/Rust boundary

Finding:

- `index.js` is correctly thin.
- `install.js` correctly owns release binary download.
- `package.json` description is broader than actual JS behavior.

### Release contract

Finding:

- `README.md`, `install.js`, and package versioning are mostly aligned.
- `verify-package.js` verifies local wrapper behavior but does not verify that the documented GitHub Release assets exist.

### Rust source boundary

Finding:

- The Rust implementation has no explicit module boundary between command parsing, config loading, pack planning, archive writing, unpack extraction, filtering, checksum, and reporting.
- The implementation is therefore easy to run but hard to use as a Rustzen reference implementation.

## PASS 3 — Unified Architecture Synthesis

### Confirmed repo status

`rustzen-zipper` remains **HIGH risk**.

The risk is not caused by product complexity. The risk is caused by all meaningful domains being concentrated in `src/main.rs`.

## Unified Rustzen Architecture Rules Update

For CLI-only Rustzen repositories:

- npm wrappers may remain thin distribution adapters.
- JS install scripts may own platform binary resolution and download logic.
- Rust CLI entry files must remain command declaration and dispatch surfaces only.
- Pack/unpack workflows count as separate command domains.
- Config resolution counts as its own application boundary.
- Filtering and path transformation count as deterministic core logic.
- Archive read/write and checksum output count as IO boundaries.
- A small CLI still qualifies as HIGH risk when one `main.rs` owns command parsing, config, workflow execution, filesystem traversal, archive IO, filtering, checksum, stats, and reporting together.

## Blog Series Article Candidate

### Designing a Local-First Packaging CLI Without Turning `main.rs` Into the Product

`rustzen-zipper` is useful because it is small. A developer can install `@rustzen/zipper`, run `rz-zip`, and get a timestamped deployment archive without adding a heavy build system. That product shape is correct for Rustzen: local-first, CLI-first, and distribution-friendly.

The architecture risk appears precisely because the product is small enough to keep growing inside one file. `src/main.rs` currently contains the public CLI contract, runtime option model, config resolution, pack workflow, unpack workflow, path filtering, archive naming, ZIP read/write, checksum output, stats collection, and terminal reporting. That makes the tool easy to ship but weak as a long-term Rustzen reference.

The useful lesson is not that every small Rust tool needs a large framework. The lesson is that command tools still need boundaries. The JS wrapper can stay thin. The npm installer can stay a distribution adapter. The Rust entry point can stay minimal. The durable product logic belongs in explicit command domains: pack, unpack, config, filtering, archive IO, and reporting.

That keeps the original purpose intact: a fast, boring, reliable Rust CLI for packaging frontend build output. The architecture target is not abstraction for its own sake. The target is to keep local-first tools maintainable after the first useful version becomes part of a real release workflow.

## Audit Delta

This pass does not modify `rustzen-zipper`.
