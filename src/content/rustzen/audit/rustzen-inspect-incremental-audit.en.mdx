## 2. PASS 1 — File-Level Inspection

### `Cargo.toml`

Observed:

- Workspace members are `zen-core`, `zen-common`, `zen-server`, and `zen-agent`.
- Workspace dependencies are explicitly SQLite-only through `sqlx` with `sqlite`, `migrate`, and `runtime-tokio-rustls` features.
- Runtime dependencies cover Axum, Tokio, cron scheduling, SQLite, zip packaging, Figment YAML config, logging, and shared utility crates.

Architecture classification:

- This repository is a multi-process Rust workspace rather than a single-service or single-CLI repo.
- The workspace-level dependency graph is aligned with Rustzen's SQLite-first rule.
- The structure is more mature than CLI-only repos, but the server and agent roots still contain large orchestration surfaces.

### `README.md`

Observed:

- Product scope is explicitly defined as a self-hosted monitoring platform.
- Runtime parts are `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common`.
- Permission rules are documented in route-local metadata terms.
- Runtime config paths and production layout are documented with concrete server/agent binaries, systemd units, SQLite runtime data, and web bundle paths.

Architecture classification:

- Documentation is operationally strong.
- The README already treats server, agent, frontend, core, and common as separate bounded roots.
- The deployment contract is clearer than the current internal startup/orchestration split.

### `docs/architecture.md`

Observed:

- The document explicitly marks the repo as a protected Web/Rust multi-process exception.
- It says the repo must not be collapsed into the generic `apps/server` + `apps/web` standard layout.
- Boundaries are explicit: server business code belongs in `zen-server/src/features/<feature>/`; agent business code belongs in `zen-agent/src/features/<feature>/` or `zen-agent/src/task/`; shared feature-neutral code belongs in `zen-core` or `zen-common`.
- Runtime facts require SQLite-only storage, YAML config, `PathResolver`, UTC across server-agent boundaries, and preservation of production runtime directories.

Architecture classification:

- The repository has one of the clearest written architecture contracts in the Rustzen set.
- It is an allowed exception to the generic Rustzen root layout.
- The remaining issue is not repository organization, but internal feature/service/task density.

### `docs/project-map.md`

Observed:

- Entrypoints are documented for server, agent, frontend, route root, layout, generated route tree, and `justfile`.
- Server feature roots include `agent`, `alert`, `auth`, `bootstrap`, `cors`, `dashboard`, `deploy`, `home`, `monitor`, `node`, `project`, `reports`, and `system`.
- Agent modules are split into `features`, `task`, `infra`, `middleware`, and migrations.
- The document also references `docs/instance-map.md` in multiple places, while the current file appears to be `docs/project-map.md`.

Architecture classification:

- The map is useful as an instance index.
- There is a documentation naming drift: architecture and README point to `docs/instance-map.md`, but the available index file is `docs/project-map.md`.
- This is a documentation consistency issue, not a runtime architecture failure.

### `docs/guides/backend.md`

Observed:

- Backend layering rule is explicit: `mod.rs` wires modules/routes only, `handler.rs` handles HTTP input/output only, `service.rs` handles orchestration/validation/transactions, `repo.rs` handles persistence only, and `types.rs` holds local DTO/row types.
- Server rules forbid services bypassing repos and repos crossing feature boundaries.
- Agent rules require local collection, local persistence, and server reporting to remain separate steps.
- Shared crate rules forbid hiding server or agent feature business logic in shared crates.
- SQLite rules prohibit non-SQLite runtime database backends.

Architecture classification:

- The backend guide is strongly aligned with Rustzen Architecture v1.
- The most important audit question is whether large `service.rs`, scheduler, and startup files stay within the documented boundaries.

### `zen-server/src/main.rs`

Observed:

- Server startup owns config load, `PathResolver`, logger initialization, directory creation, legacy runtime migration, SQLite initialization, AppState construction, CORS preload, current-project initialization, system task bootstrapping, monitor scheduler start, permission cache cleanup, route creation, permission sync, and HTTP serving.
- Task bootstrap receives database, report, alert, node, and agent services together with retention settings.
- Monitor scheduler is started directly from main after system task bootstrap.
- Permission sync is performed after route construction using registered permission codes.

Architecture classification:

- `main.rs` is not a business-domain file, but it is a broad runtime composition file.
- Startup order is explicit and readable.
- Risk: runtime lifecycle assembly, migration, scheduler startup, permission sync, and router construction all live in the entrypoint rather than a smaller boot sequence boundary.

### `zen-server/src/infra/app.rs`

Observed:

- `AppState` stores start time, server address, a database pool, and an `Arc<Services>` registry.
- Router creation merges all feature routers, static file serving, avatar resources, operation log middleware, auth middleware, CORS middleware, upload limits, and compression.
- `Services` holds a broad service registry: auth, node, bootstrap, deploy, alert, monitor, server period reports, agent, cors, project, system user, role, menu, task, dict, status, and operation log.
- `Services::new` constructs databases and services for nearly every server feature in one function.

Architecture classification:

- This is the strongest server-side concentration point.
- It is infrastructure-level composition, not domain logic, but it is large enough to become a central wiring module for almost every feature.
- Current status: PARTIAL. Acceptable as composition root, but it is near the threshold where infra assembly becomes hard to audit.

### `zen-server/src/infra/db.rs`

Observed:

- Database creation handles SQLite file creation, connection options, WAL, synchronous mode, busy timeout, mmap, cache sizing, temp storage, incremental auto-vacuum, pool creation, legacy migration-record repair, and sqlx migrations.
- The same `Database` type also exposes cleanup methods for node metrics, metric samples, alert events, operation logs, task runs, monitor probe results, and incremental vacuum.
- Cleanup uses chunked deletion to reduce write-lock duration.
- Alert-event cleanup preserves latest unresolved firing events.

Architecture classification:

- SQLite setup is explicit and operationally mature.
- `Database` is more than connection setup: it also owns cross-feature retention cleanup primitives.
- This violates the strictest form of `repo.rs handles persistence only` because infra DB now contains policy-aware cleanup behavior for multiple feature domains.

### `zen-server/src/features/system/task/service.rs`

Observed:

- `SystemTaskService` owns a task database, timezone, task catalog, and runtime scheduler.
- It registers cron jobs with `tokio-cron-scheduler`, computes next run time, persists task metadata, starts scheduler, lists tasks/runs, runs manual tasks, starts scheduled tasks, records task runs, enforces a run lock, spawns async task execution, and updates run status after execution.

Architecture classification:

- This is a valid orchestration service, but it mixes scheduler lifecycle, task registry access, task run state machine, persistence, conflict handling, and async execution management.
- It is one of the main reasons `rustzen-inspect` remains MEDIUM risk.

### `zen-server/src/features/system/task/catalog.rs`

Observed:

- Catalog defines task service dependencies, retention config, scheduled task shape, task specs, task kinds, executor mapping, cleanup targets, cleanup executors, report executor, alert sync executor, and tests for stable task schedules.
- `TASK_SPECS` encodes cleanup, report generation, and alert sync cron expressions.
- Cleanup executors call cross-feature cleanup methods directly on the infra `Database`.

Architecture classification:

- The catalog is deterministic and test-backed.
- It mixes declarative task metadata with executor construction and concrete execution behavior.
- Retention cleanup is centralized, but feature ownership is blurred because multiple feature cleanup policies are dispatched from one system task catalog.

### `zen-server/src/features/monitor/service.rs`

Observed:

- `MonitorService` owns CRUD, validation, batch create, connection testing, list/query APIs, overview, home summary, probe result recording, retention cleanup, scheduler resync, in-process scheduler startup, scheduler tick loop, immediate probe execution, probe spawning, TCP probing, and tests.
- TCP probing uses Tokio `TcpStream::connect` with timeout, matching the backend guide requirement.
- Scheduler state (`next_due_at`) is in-memory and driven by a broadcast command channel.

Architecture classification:

- Functional implementation is cohesive around monitoring, but the service layer is very broad.
- The file mixes management service, scheduler, network adapter, validation logic, and probe runner.
- This is an internal feature-level god-service risk, even though it stays within the `monitor` feature boundary.

### `zen-server/src/features/reports/server_period/service.rs`

Observed:

- `ServerPeriodService` owns listing, detail loading, period status lookup, source-data detection, metric ingestion projection, node-metric backfill, alert event ingestion, report generation, transaction state transitions, payload construction, daily rollup construction, data quality logic coordination, alert aggregation, node address normalization, and metric projection helpers.
- It works with daily/weekly/monthly/yearly report windows and uses timezone-aware period keys.
- Metric projection writes definitions and samples, computes time offset, network totals, network rates, disk usage, and derived samples.

Architecture classification:

- This service is the largest domain service pressure point in the repo.
- It is correctly located inside the `reports/server_period` feature, but it combines ingestion projection, backfill, report generation, rollup persistence coordination, alert summarization, and payload building.
- Current status: PARTIAL/FAIL at service-size level, not repository-boundary level.

### `zen-agent/src/main.rs`

Observed:

- Agent startup owns deployment preflight, config load, path resolver, logger initialization, directory creation, legacy DB migration, SQLite initialization, AppState construction, reqwest client construction, one-time node summary report, collection task start, alert config startup sync, router creation, and HTTP serving.
- `start_report_task` performs node summary reporting.
- `start_collection_task` owns collection loop creation, collector creation, scheduler creation, metrics channel creation, alert evaluation worker, local insert, aggregation run, server reporting, and alert engine dispatch.
- `report_node_metrics` performs server HTTP reporting.

Architecture classification:

- Agent `main.rs` is materially broader than server `main.rs` because it contains long-running task definitions and HTTP reporting helpers, not only startup wiring.
- This is the main agent-side boundary violation.
- The backend guide says local collection, local persistence, and server reporting should be separate steps; the runtime path does execute them in order, but the orchestration is embedded in the entrypoint.

### `zen-agent/src/infra/app.rs`

Observed:

- Agent `AppState` holds node ID, server address, metrics service, config service, health service, alerts service, and alert engine.
- `create_router` merges health, metrics, config, deploy, and alerts routers, applies auth middleware, upload limits, compression, and state.

Architecture classification:

- Agent app composition is smaller than server app composition.
- It remains acceptable as infrastructure wiring.
- It also reveals that runtime task startup is not part of `infra/app.rs`, because task startup currently lives in `main.rs`.

### `zen-agent/src/task/scheduler.rs`

Observed:

- `Scheduler` owns config, aggregator, cleaner, SQLite pool, cleanup timezone, and last cleanup day.
- `run_aggregation_once` runs minutely, hourly, daily, monthly aggregation and cleanup in one call.
- Aggregation existence is checked with direct SQL inside scheduler.
- Cleanup timing is timezone-aware and test-backed.
- Tests verify once-per-day cleanup behavior, configured timezone handling, and no catch-up when starting after the configured time.

Architecture classification:

- Scheduler is deterministic and operationally practical.
- It mixes orchestration with direct persistence queries and cleanup timing state.
- The direct SQL existence check is a small repo-layer leak inside a task scheduler.

### `zen-agent/src/task/aggregator.rs`

Observed:

- Aggregator owns SQL reads, raw-to-aggregate conversion, aggregate-to-aggregate conversion, CPU/load/memory/swap/disk/network/process/uptime computations, JSON parsing, active network filtering, and upsert coordination.
- It defines a long SQL projection for aggregated metrics and holds complex metric math in one module.
- Disk aggregation and network aggregation are internal helper domains within the same file.

Architecture classification:

- This is the main agent-side computation god module.
- It is not misplaced at the repo level because aggregation belongs in `zen-agent/src/task/`, but the module combines persistence, domain math, JSON decoding, filtering, and upsert behavior.
- Current status: PARTIAL/FAIL at task-module boundary level.

## 3. PASS 2 — Cross-Module Consistency

### Documentation vs actual file names

Finding:

- README and architecture docs refer to `docs/instance-map.md`.
- The current index file available in the repository is `docs/project-map.md`.

Classification:

- Documentation consistency issue.
- Does not affect runtime, but weakens deterministic agent navigation because the declared map path does not match the actual file path.

### Backend guide vs server implementation

Finding:

- The guide says `handler.rs` is HTTP-only, `service.rs` is orchestration/validation/transactions, and `repo.rs` is persistence-only.
- Many feature directories broadly follow this shape.
- However, server infra DB owns retention cleanup across multiple features, system task catalog owns executor behavior, and large services such as `MonitorService` and `ServerPeriodService` combine multiple internal subdomains.

Classification:

- Repository-level boundary: PASS.
- Internal service/task granularity: PARTIAL.

### Server task scheduler vs report/monitor scheduling

Finding:

- System tasks are centralized through `SystemTaskService` and `TaskCatalog`.
- Monitor probing has a separate in-process scheduler inside `MonitorService`.
- Agent aggregation/cleanup has another scheduler inside `zen-agent/src/task/scheduler.rs`.

Classification:

- This is not necessarily wrong because the repo spans server and agent processes.
- The architectural risk is having multiple scheduler patterns with different ownership models: catalog-backed persisted tasks, monitor in-memory probes, and agent local aggregation/cleanup.

### Agent guide vs agent main

Finding:

- The guide requires local collection, local persistence, and server reporting as separate steps.
- The runtime implementation performs those steps sequentially, but the collection loop, persistence call, aggregation call, reporting call, and alert dispatch are all orchestrated from `zen-agent/src/main.rs`.

Classification:

- Behavior aligns with the guide.
- Code placement does not fully align with Rustzen's entrypoint-thinness rule.

### Shared crate boundary

Finding:

- `zen-core` and `zen-common` are documented as shared primitives/utilities, not business logic containers.
- This pass did not observe server/agent feature business logic being hidden in shared crates.

Classification:

- PASS for shared crate boundary in reviewed files.

## 4. PASS 3 — Unified Architecture Synthesis

### Confirmed repo status

`rustzen-inspect` remains **MEDIUM risk**.

It is not MEDIUM because of poor repository structure. It is MEDIUM because mature repository boundaries contain several large internal execution modules.

### Risk profile

| Area | Status | Notes |
|------|--------|-------|
| Workspace layout | PASS | Multi-process exception is explicit and protected |
| SQLite-first rule | PASS | Workspace `sqlx` config is SQLite-only |
| Documentation | PARTIAL | Strong docs, but `instance-map.md` vs `project-map.md` drift exists |
| Server app composition | PARTIAL | `infra/app.rs` is large but mostly composition-root logic |
| Server entrypoint | PARTIAL | Startup sequence is explicit but broad |
| System task scheduler | PARTIAL | Scheduler lifecycle, run state, persistence, and executor dispatch are combined |
| System task catalog | PARTIAL | Deterministic task specs, but specs and executors are mixed |
| Monitor feature | PARTIAL | CRUD + scheduler + probe adapter + validation live in one service file |
| Server period reports | FAIL | Large service combines ingestion projection, backfill, report generation, alert aggregation, payload construction |
| Agent entrypoint | FAIL | Startup plus long-running collection/reporting task orchestration live in `main.rs` |
| Agent scheduler | PARTIAL | Aggregation/cleanup orchestration includes direct SQL existence checks |
| Agent aggregator | FAIL | SQL, JSON parsing, metric math, filtering, and upsert coordination are combined |
| Shared crates | PASS | Reviewed files do not hide feature business logic in shared crates |

### Architecture delta from this pass

This pass refines the existing Rustzen rule for multi-process products:

- A multi-process repository may keep protected roots such as `zen-server`, `zen-agent`, `zen-web`, `zen-core`, and `zen-common` when the deployment model depends on them.
- Passing the repository-layout rule is not enough; each process still needs thin entrypoints and bounded runtime services.
- Scheduler code must declare ownership clearly: persisted system task scheduler, in-memory probe scheduler, and local agent aggregation scheduler are allowed to coexist only when their state, persistence, and execution semantics are explicit.
- `service.rs` may orchestrate workflows, but it should not become the only home for validation, scheduling, network adapters, metric projection, backfill, payload construction, and persistence coordination.
- `infra/db.rs` may initialize SQLite and migrations, but feature-specific retention policy should not become hidden database-infra behavior unless it is explicitly documented as cross-feature infrastructure.
- Agent `main.rs` should remain startup and dispatch only; long-running collection/persistence/reporting/alert loops should be auditable as task-level workflows.
- Aggregation modules are allowed to be computation-heavy, but metric math, SQL access, JSON decoding, active network filtering, and upsert behavior should remain distinguishable inside the module.

|------|---------------|--------------|-------|
| rustzen-inspect | MEDIUM | MEDIUM | Confirmed; internal task/service density is the key risk |

## 5. Per-Repo Consistency Check

`rustzen-inspect` is stronger than `rustzen-zipper`, `rustzen-clear`, and `rustzen-clipboard` on written architecture and repository boundaries.

The main architectural pressure points are not missing layers, but overloaded internal modules:

1. `zen-server/src/infra/app.rs` as a broad composition root.
2. `zen-server/src/features/system/task/service.rs` as scheduler + task-run state + execution boundary.
3. `zen-server/src/features/system/task/catalog.rs` as task registry + executor factory + execution implementations.
4. `zen-server/src/features/monitor/service.rs` as monitor management + scheduler + TCP adapter.
5. `zen-server/src/features/reports/server_period/service.rs` as report ingestion + generation + payload builder.
6. `zen-agent/src/main.rs` as startup + runtime task orchestration.
7. `zen-agent/src/task/scheduler.rs` as aggregation + cleanup coordinator with direct SQL checks.
8. `zen-agent/src/task/aggregator.rs` as persistence + JSON decoding + metric calculation + upsert coordinator.

## 6. Unified Rustzen Architecture Rules Update

Incremental rule update from this repository only:

### Rule: Multi-process exception must not become service-density exception

A repository may keep separate process roots when deployment requires them. That exception applies to root layout only. It does not waive the core Rustzen rule that entrypoints stay thin and services remain bounded.

### Rule: Scheduler ownership must be explicit

For Rustzen repositories with multiple schedulers, each scheduler must make its state model clear:

- persisted task scheduler
- in-memory runtime scheduler
- local agent scheduler

They may coexist, but they must not hide persistence state, retry semantics, cleanup policy, and execution dispatch in indistinguishable service methods.

### Rule: Retention policy is domain policy

Chunked deletion and SQLite maintenance are persistence operations. Retention windows, preservation exceptions, and per-feature cleanup targets are domain policy. If retention is centralized, the owning boundary must be explicit in audit docs.

### Rule: Monitoring adapters are infrastructure-adjacent

TCP probing, timeout handling, and connection-test runners are adapter logic. They may be colocated with a feature during early product growth, but audit classification must treat them as separate from CRUD service behavior.

### Rule: Metric aggregation must expose sub-boundaries

Metric aggregation may stay in a task module, but SQL fetching, JSON decoding, network filtering, metric math, rollup calculation, and upsert persistence must remain auditable as separate responsibility groups.
