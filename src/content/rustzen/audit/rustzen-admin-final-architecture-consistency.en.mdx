### Rustzen Architecture v1 update

Add this audit rule:

> Web/Rust A-class layout is not a full architecture score. A repository can be the reference layout and still carry MEDIUM implementation risk when operational runtime features concentrate startup migration, scheduler lifecycle, deploy artifact validation, symlink activation, rollback, service restart, and static/runtime serving inside a small number of files.

Add this consistency rule:

> In Rustzen Architecture v1, admin repositories must be evaluated on two axes: layout/documentation conformance and operational-runtime density. `api -> service -> repo` remains the baseline for CRUD-style admin features, but deploy management, startup migration, scheduled tasks, and root auth/permission routing are runtime ownership surfaces and must be audited separately.

### Repo-specific final check

- Do not treat `rustzen-admin` as layout migration debt.
- Do not classify it as LOW risk only because the workspace layout is clean.
- Do not score deployment management as ordinary CRUD.
- Score it as a local-first admin/runtime foundation with strong documentation and medium operational density.

## Blog Series Article

### Designing a Local-First Admin Runtime Without Hiding the Operational Core

An admin product usually looks like a set of tables: users, roles, menus, dictionaries, logs, scheduled tasks, and deployment versions. That view is useful, but incomplete. In a local-first Rust admin runtime, the real architecture is not only the CRUD layer. The real architecture is the boundary between ordinary management screens and operational behavior that changes the running system.

`rustzen-admin` is a useful reference because its repository layout is clear. The backend lives in `apps/server`, the frontend lives in `apps/web`, shared Rust capability crates live under `crates/*`, deployment assets live under `deploy`, and current architecture rules live under `docs/`. This shape gives AI-assisted development a stable map. A contributor can usually find where a change belongs before touching code.

The important lesson is that a clean layout does not remove runtime pressure. Deployment management is not just a table of versions. It parses uploaded packages, validates server binaries and web zips, hashes content, verifies markers and signatures, writes versioned artifacts, switches symlinks, activates web dist output, updates the current deployment record, rolls back failed activation, and schedules process restart. That is product behavior, not generic admin plumbing.

The same pattern appears in scheduled tasks. A task page can look simple in the UI, but the backend owns cron parsing, timezone interpretation, fixed task catalog synchronization, stale run cleanup, run locking, async task spawning, executor dispatch, and completion persistence. This is a runtime lifecycle, not just a list endpoint.

The correct Rustzen reading is therefore two-layered. First, check whether the repository has a stable shape that humans and AI can follow. `rustzen-admin` passes that test. Second, check where runtime ownership concentrates. In this repo, the main pressure points are startup assembly, database/runtime migration, scheduled task execution, and deploy activation.

That distinction matters for the Rustzen Architecture v1 spec. A local-first product should avoid pretending that operational behavior is ordinary CRUD. The code can still be pragmatic and compact, but the audit language must name the real ownership surface. For `rustzen-admin`, the architecture value is not that everything is abstracted away. The value is that the product keeps a readable Web/Rust reference layout while making the operational core visible enough to review.

The product narrative is also consistent with the broader Rustzen direction. Rustzen tools should be understandable from source, runnable locally, concrete in runtime paths, and conservative about abstractions. `rustzen-admin` shows the strength and the cost of that approach: strong boundaries at the repository level, with careful attention still required wherever the code crosses from management UI into runtime-changing behavior.
