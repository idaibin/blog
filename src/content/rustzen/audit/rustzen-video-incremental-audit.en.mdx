# Incremental Audit: rustzen-video

## Scope

Repository reviewed in this pass: `rustzen/rustzen-video`.

Reviewed files:

- `README.md`
- `Cargo.toml`
- `src/main.rs`
- `src/infra/config.rs`
- `src/infra/logger.rs`
- `src/shared/error.rs`
- `src/features/video/api.rs`
- `src/features/video/service.rs`
- `src/features/video/dto.rs`
- `src/features/video/state.rs`
- `src/features/video/pipeline.rs`
- `src/features/video/flv.rs`
- `src/features/video/ws.rs`
- `src/features/video/rtsp_resolver.rs`

## PASS 1 — File-Level Inspection

### `README.md`

Observed:

- The repository is positioned as an RTSP to WebRTC gateway with HTTP API and frontend playback surface.
- The documented target layout is compact but explicit: `main.rs`, `infra`, `features/video`, and `shared/error`.
- The document states that root `AppState` should only contain global config and `VideoState`.
- The video feature is expected to contain API, DTO, state, pipeline, FLV, WebSocket, and RTSP resolver modules.
- Deployment notes are operationally detailed and include Docker, offline package export, Kylin/Rocky Linux checks, GStreamer plugin checks, WebRTC UDP port mapping, RTSP retry settings, and diagnostic commands.

Architecture classification:

- PASS for intent clarity.
- PARTIAL for documentation hygiene.
- The top-level architecture map is useful, but the README also contains environment-specific debug notes and internal network examples. This creates documentation density around deployment/debugging rather than product architecture.

### `Cargo.toml`

Observed:

- Single crate named `rustzen-video`, version `0.1.0`, edition `2024`.
- Runtime dependencies include Axum, Tokio, GStreamer, GStreamer WebRTC/App/SDP, reqwest, tower-http, tracing, uuid, sysinfo, and thiserror.
- Tokio features are limited to runtime, macros, net, and sync.
- HTTP, WebSocket, video pipeline, external resolver, logging, and process metrics all live in one crate.

Architecture classification:

- MEDIUM risk.
- Dependency surface is expected for a media gateway, but all runtime domains are currently concentrated in one binary crate.
- This is not a workspace-boundary failure; it is a runtime-domain density issue.

### `src/main.rs`

Observed:

- Defines root `AppState { config, video }`, matching README intent.
- Initializes logger and exits the process on startup failure.
- Initializes GStreamer directly in `run`.
- Checks required GStreamer plugin factories during startup.
- Loads config, creates `VideoState`, assembles the video routes, serves static `web`, binds TCP, and starts Axum.

Architecture classification:

- PASS for small root state.
- PARTIAL for app assembly pressure.
- `main.rs` currently owns process bootstrap, GStreamer environment validation, config loading, feature state creation, route assembly, static serving, and network binding.
- The pressure is localized and acceptable for a focused gateway, but it is not a pure entrypoint.

### `src/infra/config.rs`

Observed:

- Loads only `PORT` from environment.
- Defaults to `9898`.
- Returns a minimal `AppConfig`.

Architecture classification:

- PASS.
- Config scope is small and deterministic.
- The file does not absorb feature-specific policy.

### `src/infra/logger.rs`

Observed:

- Creates a `logs` directory.
- Configures daily rolling file logging with `MAX_LOG_FILES = 7`.
- Installs stdout and file logging layers.
- Returns `WorkerGuard` to keep the non-blocking writer alive.

Architecture classification:

- PASS.
- Logging is correctly isolated as infrastructure.
- Retention policy is fixed and explicit.

### `src/shared/error.rs`

Observed:

- Defines `AppError` for validation, bad request, conflict, not found, unauthorized, service unavailable, and internal errors.
- Implements `IntoResponse` with JSON `{ code, error }`.
- Logs internal errors and hides internal details from API responses.
- Several constructors and variants are allowed as dead code.

Architecture classification:

- PASS-PARTIAL.
- The boundary is correct: API-facing error mapping is centralized.
- The dead-code allowance indicates this is a reusable error surface larger than current handlers need.

### `src/features/video/api.rs`

Observed:

- Defines `/api-rtc/ws`, `/api-rtc/status`, and `/api-rtc/flv`.
- Status handler creates `VideoService` from shared `AppState`.
- Router remains thin and does not directly create pipelines.

Architecture classification:

- PASS.
- Handler layer is clean and thin.
- Feature routes do not leak static serving or global app assembly into the video module.

### `src/features/video/service.rs`

Observed:

- `VideoService` wraps `Arc<AppState>` rather than `Arc<VideoState>`.
- `status` maps runtime state snapshots into DTOs and reads current process memory.
- `resolve_rtsp` delegates to `rtsp_resolver`.
- WebSocket error mapping and resolver error classification are handled in the service.

Architecture classification:

- MEDIUM localized risk.
- Service is not oversized yet, but it mixes status projection, RTSP input resolution, and WebSocket error taxonomy.
- The most important boundary concern is that service depends on the root app state instead of the feature state plus explicit collaborators.

### `src/features/video/dto.rs`

Observed:

- HTTP status response DTOs are separated from state types.
- WebSocket client and server messages are defined as tagged serde enums.
- WebRTC and FLV protocol concepts share the same DTO module.

Architecture classification:

- PASS.
- Transport contracts are explicit.
- The module is acceptable because it remains data-only.

### `src/features/video/state.rs`

Observed:

- `VideoState` owns WebRTC stream sources, WebSocket connections, session senders, RTSP metrics, FLV shared sources, FLV build locks, `sysinfo::System`, and reused-source counters.
- It uses several `std::sync::Mutex` containers plus one async mutex map for FLV build locks.
- It contains runtime status projection helpers, RTSP metric mutation, session sender registration, FLV lock management, process memory sampling, and counters.
- It explicitly documents non-atomic status snapshots.

Architecture classification:

- MEDIUM-HIGH risk.
- This is the main state concentration point.
- `VideoState` is not just state storage; it also contains runtime metrics policy, lock lifecycle, snapshot projection, process metrics, and FLV coordination helpers.
- The risk is concurrency and lifecycle clarity, not missing modules.

### `src/features/video/pipeline.rs`

Observed:

- Contains RTSP retry environment parsing, recoverable-error detection, user-facing error mapping, offer negotiation, stream source lifecycle, GStreamer parse-launch construction, bus watcher threads, WebRTC branch creation, FLV branch creation, FLV tag filtering, branch removal, ICE handler connection, and tests.
- WebRTC and FLV pipeline assembly both spawn bus-watch threads.
- `StreamSource::drop` flips a stop flag and spawns a thread to set the pipeline state to `Null`.
- FLV body pumping uses an appsink thread and drops chunks when the bounded channel is full.
- Tests cover retry classification and FLV tag helpers.

Architecture classification:

- HIGH localized risk.
- This is the highest-density file in the repository.
- It combines media pipeline creation, retry policy, error taxonomy, protocol negotiation, lifecycle shutdown, thread ownership, FLV parsing, and WebSocket ICE dispatch support.
- The file is functionally coherent around media pipelines, but it is carrying multiple runtime policies at once.

### `src/features/video/flv.rs`

Observed:

- Handles FLV WebSocket upgrade, RTSP input resolution, GStreamer-vs-ffmpeg selection, ffmpeg process spawning, stdout/stderr pumping, shared source reuse, build locks, broadcast fanout, reconnect policy, and client streaming.
- `FlvGuard` kills ffmpeg children on drop and holds GStreamer sources for drop-based shutdown.
- Shared FLV streams are keyed by normalized RTSP URL plus format.
- Reconnect behavior is implemented inside the WebSocket streaming handler.

Architecture classification:

- HIGH localized risk.
- The module is larger than a handler and larger than a pure transport adapter.
- It contains process management, source caching, concurrency control, stream fanout, reconnect policy, and client socket handling in one place.
- This is acceptable as an early focused gateway, but it is the second major runtime-density point after `pipeline.rs`.

### `src/features/video/ws.rs`

Observed:

- Handles WebRTC WebSocket upgrade, connection registration, writer task, read loop, message parsing, session creation, RTSP resolution, pipeline creation, offer send, ICE registration, answer handling, ICE handling, disconnect, disconnect all, and cleanup.
- It directly mutates `VideoState` connection maps, stream source maps, and session sender maps.
- Cleanup logic is careful about lock release before session drop.
- Message parsing and error responses are explicit.

Architecture classification:

- MEDIUM-HIGH risk.
- WebRTC transport handling is coherent but dense.
- The strongest positive point is cleanup discipline.
- The main boundary concern is that transport code also performs orchestration, session lifecycle, stream registration, and pipeline ownership.

### `src/features/video/rtsp_resolver.rs`

Observed:

- Accepts either direct `rtsp://` input or a camera code.
- Direct RTSP input returns immediately.
- Camera-code mode calls a fixed internal HTTP endpoint.
- The resolver maps external API structure into `ResolvedRtsp`.
- Timeout is hardcoded to 8 seconds.

Architecture classification:

- MEDIUM risk.
- The resolver is correctly isolated as video-feature logic.
- The hardcoded internal endpoint makes this repository environment-coupled.
- This is not a Rust layer violation, but it is an operational boundary risk.

### `web/index.html`

Observed:

- Static frontend is a single HTML file with CSS and JavaScript inlined.
- It owns camera list loading, camera filtering, 4-slot layout, WebSocket management, WebRTC answer flow, FLV player lifecycle, status messages, session metadata, and reconnect/error handling.
- It fetches `/webrtc/camera-list.json` for local camera list data.
- FLV playback uses `flv.min.js` from a local script path.

Architecture classification:

- MEDIUM risk.
- For a demo UI this is acceptable, but it is not a maintainable frontend boundary.
- The frontend mirrors backend session complexity inside one file.
- The file should be treated as a demo playback client, not a Rustzen frontend baseline.

## PASS 2 — Cross-Module Consistency

### Documentation vs implementation

Consistent:

- README documents the expected compact layout.
- Actual source follows `infra`, `shared`, and `features/video` boundaries.
- Root `AppState` contains only config and video state, matching the documentation.

Finding:

- Repository-level layout is stronger than the internal runtime boundaries.
- The main architecture pressure appears inside feature modules, not at the root layout level.

### Handler/service/state boundary

Consistent:

- `api.rs` remains thin.
- `service.rs` absorbs status projection and resolver error classification.
- `state.rs` centralizes shared runtime state.

Finding:

- The service boundary is present but shallow.
- Most important behavior is still in `ws.rs`, `flv.rs`, `pipeline.rs`, and `state.rs`.

### WebRTC and FLV boundary

Consistent:

- WebRTC socket flow lives in `ws.rs`.
- FLV socket flow lives in `flv.rs`.
- Both converge on `pipeline.rs` and `VideoState`.

Finding:

- Separation by protocol exists, but shared lifecycle and pipeline responsibilities converge in two dense places: `pipeline.rs` and `VideoState`.

### Runtime lifecycle boundary

Consistent:

- `StreamSource::drop` attempts to shut down GStreamer pipelines.
- FLV guards kill ffmpeg children.
- WebSocket close paths drop sessions and remove stream sources.

Finding:

- Lifecycle cleanup exists and is intentional.
- Ownership is still distributed across drop impls, spawned threads, WebSocket tasks, broadcast channels, state maps, and process guards.

### Product/documentation boundary

Consistent:

- The repo is clearly a real functional gateway rather than a pure prototype.
- README has practical deployment notes.
- Frontend remains a demo-style static client.

Finding:

- The backend is closer to product behavior than the frontend.
- The frontend should remain classified as demo surface in audit output.

## PASS 3 — Unified Architecture Synthesis

### Confirmed repo status

`rustzen-video` is classified as **MEDIUM-HIGH risk**.

Reason:

- Top-level layout is clear and compact.
- API handlers are thin.
- Config, logger, and error handling are reasonably isolated.
- Risk concentrates in media runtime orchestration: `pipeline.rs`, `flv.rs`, `ws.rs`, and `VideoState`.
- The repository is small enough to understand, but runtime ownership is spread across GStreamer elements, ffmpeg child processes, WebSocket tasks, broadcast channels, state maps, and drop-based cleanup.

### Standard refinement from this pass

For Rustzen media gateway repositories:

- Media runtime boundaries must be audited separately from ordinary `api -> service -> repo` layering.
- A thin route layer does not prove clean architecture when protocol handlers own long-lived runtime tasks.
- `State` modules must not be treated as neutral containers when they also own lifecycle helpers, metrics policy, locks, counters, and process sampling.
- Pipeline modules must be reviewed as runtime engines when they own media construction, retry policy, protocol negotiation, bus-watch threads, error taxonomy, and shutdown behavior.
- Demo static clients must not be treated as Rustzen frontend baselines when they embed all UI, protocol, session, and playback logic in one file.
- Hardcoded environment-specific resolver endpoints are operational coupling and must be tracked in audit output.

## Audit Delta

This pass adds a media-gateway-specific Rustzen rule:

> A Rustzen video/media gateway cannot be evaluated only by file layout or thin HTTP handlers. Any module that owns long-lived media pipelines, external process handles, protocol negotiation, broadcast fanout, reconnect policy, and drop-based shutdown must be treated as a runtime engine boundary in audit output.

No source repository was modified in this pass. Only this audit document was added under `rustzen/blog`.

## Blog Series Candidate

### Designing a Local-First Video Gateway Without Losing Runtime Boundaries

A local-first video gateway looks simple from the outside: receive an RTSP source, expose a browser-friendly stream, and keep the deployment self-contained. `rustzen-video` follows that direction with a compact Rust server, GStreamer media pipelines, WebRTC signaling, FLV fallback, and a static playback client.

The useful architectural lesson is that media systems do not become simple just because their repository layout is simple.

The top-level structure is clear: infrastructure handles config and logging, the video feature owns the media domain, and shared error handling keeps API responses consistent. The route layer is also intentionally thin. This is the correct starting point for a focused gateway.

The real pressure appears below the route layer. Media gateways have long-lived runtime objects: GStreamer pipelines, WebRTC elements, ffmpeg child processes, WebSocket sessions, broadcast fanout channels, reconnect loops, and cleanup paths. These objects do not behave like ordinary request-response services. They create ownership problems.

`rustzen-video` shows this clearly. `pipeline.rs` is not just a helper file. It is a runtime engine: it builds media graphs, negotiates WebRTC offers, watches GStreamer bus messages, filters FLV frames, maps errors, retries RTSP sources, and shuts down pipelines through drop behavior. `flv.rs` is not just an HTTP handler. It manages ffmpeg processes, shared stream reuse, broadcast delivery, reconnect attempts, and client WebSocket output. `ws.rs` is not just signaling glue. It owns connection state, session creation, stream registration, ICE handling, answer application, and cleanup.

The core Rustzen rule from this review is straightforward: media runtime ownership must be reviewed as a first-class architecture boundary. A clean `api -> service -> state` shape is not enough when runtime tasks continue after the original request is gone.

This does not mean the gateway is badly structured. It means the audit category is different from an ordinary admin server or local database app. The correct question is not only “are handlers thin?” but also “who owns the pipeline, who stops it, who records failure, who fans out bytes, who reconnects, and who cleans up when the client disappears?”

For Rustzen Architecture v1, `rustzen-video` becomes the reference case for media-runtime review. It confirms that local-first Rust services can stay compact, but compactness must not hide lifecycle responsibility.

The result is a practical rule for future Rustzen media tools: keep the repository small, but make runtime ownership visible.
