- The status block records `release: none`, `updater: none`, and root `src-tauri` as a prototype-only layout exception.

Classification:

- Product scope: PASS as prototype.
- Release posture: PASS as blocked/non-release.
- Template posture: PASS as explicitly non-template.
- Architecture risk: LOW-MEDIUM at repository level because the prototype boundary is documented, but runtime ownership remains under-specified.

Classification:

- Clipboard monitor: MEDIUM-HIGH local runtime risk.
- Event contract: incomplete.
- Shutdown ownership: incomplete.

## Rustzen Architecture v1 Rule Update

### Rule: Launcher prototypes are not product templates

A Rustzen launcher-style Tauri repository may keep a root `src-tauri` layout only when all of the following are true:

1. The repository is explicitly classified as a prototype.
2. Release status is `none`.
3. Updater status is `none`.
4. Bundling is disabled or explicitly non-release.
5. Documentation states that the layout must not be copied into new desktop products.
6. Mock UI and missing command/data contracts are treated as blockers, not product features.

### Rule: Shortcut surfaces require runtime scoring even when product scope is small

A small desktop launcher still requires runtime-engine scoring when it owns any of the following:

- global shortcut registration;
- background clipboard polling;
- Tauri plugin setup;
- event emission from Rust to frontend;
- window visibility and size lifecycle;
- manual Tokio runtime creation.

This repository remains LOW-MEDIUM because those behaviors exist but are contained by the documented prototype boundary.

## Rustzen Blog Series Article

### Designing a Local-First Launcher Prototype Without Turning It Into a Template

A launcher looks small from the outside. It has an input box, a keyboard shortcut, a few search results, and a window that opens quickly. That surface can make the product feel simple, but the implementation already crosses several runtime boundaries.

`rustzen-launcher` is useful precisely because it keeps that distinction visible. It is not presented as a finished desktop product. It is not presented as the default Rustzen desktop layout. It is a prototype for exploring a launcher/search surface.

That matters for Rustzen Architecture v1 because a prototype can contain direct runtime wiring without pretending that the wiring is reusable architecture. In this repository, Tauri startup, plugin registration, global shortcuts, clipboard polling, and window resizing are all present. Those are real runtime behaviors, even if the search data is still mock data.

The main lesson is that local-first does not mean structure-free. A local launcher still needs clear ownership for background tasks, event names, window lifecycle, and command execution. Until those contracts exist, the repository should remain a prototype.

This is why the root `src-tauri` layout is acceptable here but not as a general Rustzen desktop rule. The exception is bounded by documentation: no release baseline, no updater, no signing workflow, no bundling posture, and no claim that other Rustzen desktop products should copy this structure.

The better architecture signal is restraint. `rustzen-launcher` does not invent a platform. It does not add cloud services. It does not introduce a generic plugin system. It records the current exploration surface and keeps the release boundary closed.

For Rustzen, that is the correct use of a prototype: explore the interaction model, expose the runtime pressure early, and prevent the prototype from silently becoming the template.
