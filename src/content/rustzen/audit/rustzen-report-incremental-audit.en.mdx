# Incremental Audit: rustzen-report

## Scope

Repository reviewed in this pass: `rustzen/rustzen-report`.

Reviewed files:

- `Cargo.toml`
- `README.md`
- `docs/architecture.md`
- `docs/project-map.md`
- `docs/guides/backend.md`
- `zen-server/src/main.rs`
- `zen-server/src/app.rs`
- `zen-server/src/features/job/service.rs`
- `zen-server/src/features/job/scheduler.rs`
- `zen-server/src/features/job/executor.rs`
- `zen-server/src/features/job/repo.rs`

## PASS 1 — File-Level Inspection

### `Cargo.toml`

Observed:

- Workspace contains a single Rust member: `zen-server`.
- Workspace dependencies combine Axum, SQLx SQLite, Tokio, Chromium automation, request clients, tracing, UUID, dotenv, and timezone support.
- Release profile is size-oriented with `opt-level = "z"`, thin LTO, single codegen unit, abort panic, and stripping.

Architecture classification:

- Rust workspace boundary is simple and deterministic.
- The repository is not a multi-crate Rust library system; the backend pressure is concentrated inside one server crate.
- Dependency set confirms this is a browser-driven report automation backend, not a pure CRUD backend.

### `README.md`

Observed:

- Defines `rustzen-report` as a full-stack report system with Rust backend and React frontend.
- Runtime state is local-first: SQLite database, uploads, frontend static files, and logs live under repository-defined runtime paths.
- Documents backend modules: auth, account, user, system, flow, dataset, template, job, cron, periodic, health, overview, uploads, and websocket.
- Defines deployment contract around `/opt/rustzen-report`, `HOME_PATH`, `data/report-server.db`, `data/uploads/`, `web/dist/`, and `logs/`.
- Notes a startup migration for legacy root-level SQLite file groups into `data/report-server.db*`.

Architecture classification:

- Documentation is current and operational.
- Runtime layout is aligned with Rustzen local-first expectations.
- The README correctly describes the system as a full-stack application, not a reusable template.

### `docs/architecture.md`

Observed:

- Classifies the repository as `Web/Rust A-class legacy/current layout`.
- Protects the current `zen-server` and `zen-web` structure and explicitly prevents general migration to `apps/server` + `apps/web`.
- Defines second-level backend layout as `zen-server/src/features/<feature>/` and `zen-server/src/infra/<subsystem>/`.
- Defines protected deployment contract: `target/rustzen-report`, `/opt/rustzen-report`, `bin/report-server`, `report-server.service`, `deploy/install.sh`, and `HOME_PATH=/opt/rustzen-report`.
- Requires API, database, directory, docs, and command entrypoints to change together.

Architecture classification:

- PASS at documentation boundary level.
- The repository has clear AI-facing rules and protected layout language.
- The main audit concern is not missing documentation; it is high-density workflow implementation inside the job execution path.

### `docs/project-map.md`

Observed:

- Maps backend startup to `zen-server/src/main.rs`, app/router assembly to `zen-server/src/app.rs`, and shared command entrypoint to `justfile`.
- Identifies job execution, cron tasks, periodic generation, overview read-model, uploads, and websocket as distinct backend features.
- Separates frontend route pages, API wrappers, shared state, reusable components, and generated route files.
- Marks high-frequency backend change points: feature endpoints, auth guards, database access, deployment/runtime rules, and permission rules.

Architecture classification:

- Project map is useful and mostly consistent with code layout.
- It records the same architectural pressure visible in implementation: job execution and app assembly are high-frequency, high-density paths.

### `docs/guides/backend.md`

Observed:

- Defines feature file roles: `mod.rs` route assembly, `api.rs` request/response, `service.rs` orchestration/validation/transactions, `repo.rs` SQL/persistence, and `types.rs` local types.
- Prohibits SQL in `api.rs`, service bypass of repos, repo cross-feature calls, compatibility fallbacks, generated-output edits, and speculative abstraction.
- Defines `overview` as the only current read-model exception.
- Keeps `JobRepository::rename_job_id_columns` as a narrow runtime schema compatibility exception.
- Allows report job target URLs to point to intranet systems and therefore avoids generic private-IP blocking.

Architecture classification:

- Strong architecture guide.
- The guide is stricter than parts of the current job implementation.
- The narrow compatibility exception is explicit and bounded, but it must remain exceptional.

### `zen-server/src/main.rs`

Observed:

- Startup entrypoint is thin: loads `.env`, reads config, initializes timezone before Tokio runtime, builds runtime, then delegates to `run`.
- `run` ensures runtime directories, initializes logging, migrates legacy SQLite files, creates pool, runs migrations, constructs app state, starts cron background work, builds router, binds listener, and serves Axum.
- Timezone mutation is guarded by validation through `chrono_tz` and is performed before runtime/background workers are created.

Architecture classification:

- PASS as an entrypoint.
- Runtime sequence is explicit and deterministic.
- Startup still performs several operational actions, but most composition pressure is delegated to `AppState` and router assembly.

### `zen-server/src/app.rs`

Observed:

- `AppState` directly owns config, database pool, task registry, auth service, job service, template service, overview service, periodic service, cron service, uploads service, user/system/account/flow/dataset services, and websocket hub.
- `AppState::new` constructs repositories and services, wires cross-feature dependencies, builds the job service dependency object, registers periodic cron task, and creates overview/user services.
- `create_router` merges protected API routes for auth, health, websocket, job, overview, cron, uploads, user, system, account, flow, and dataset.
- `auth_middleware` is colocated in app assembly and manually exempts auth, health, and job websocket paths.
- SPA static serving is mounted in the same router assembly file.

Architecture classification:

- MEDIUM risk.
- This is mainly composition pressure, not business logic leakage.
- The file is still a central lifecycle hub: dependency graph, scheduled task registration, route assembly, auth middleware, CORS, tracing, and static fallback all meet in one place.

### `zen-server/src/features/job/service.rs`

Observed:

- `JobService` owns registry, repo, template, flow, system, account, dataset, upload, config, and websocket dependencies.
- `normalize_create_job_request` resolves flow/system/dataset/account consistency, request metadata, passthrough records, dataset bundles, and report execution dimensions.
- `create_job` resolves template metadata, generates request id, handles existing runtime overwrite, clears old results, registers websocket secret, creates runtime, persists summary, and spawns the job.
- `prepare_job` and `resolve_job_resources` both perform system-template validation, dataset resolution, account selection, and execution planning/resource projection.
- Later helpers in the same file parse/merge logs, merge runtime summaries, build execution plans, normalize dataset ids/row numbers, merge dataset rows, map rows into records, paginate memory results, and resolve period filters.

Architecture classification:

- HIGH localized risk.
- This is the densest service file observed in this repository.
- The file is still within a feature boundary, but it mixes job lifecycle, request normalization, dataset planning, resource projection, runtime state reconciliation, log serialization, period filtering, and row-shape normalization.

### `zen-server/src/features/job/scheduler.rs`

Observed:

- `JobSpawnInput` carries runtime, registry, repo, uploads, template, system, account, dataset, config, and websocket dependencies.
- `spawn_job` acquires concurrency permit, validates system id, resolves fill template, prepares rows, converts rows to records, resolves accounts and secrets, executes the browser job, persists summary, emits websocket finish/error events, and removes channels.
- Error handling is explicit but repeated across missing system, missing template, setup failure, and terminal state publishing.
- `convert_rows_to_records` is small but lives in scheduler, while row-shape normalization also appears in service-level helpers.

Architecture classification:

- MEDIUM-HIGH risk.
- Scheduler owns more than scheduling; it performs setup orchestration, data shaping, account resolution, execution dispatch, persistence, and websocket finalization.
- The file remains understandable, but it is not a pure scheduler boundary.

### `zen-server/src/features/job/executor.rs`

Observed:

- Executor imports browser/CDP protocol types, path safety, base64, uploads, websocket events, template rendering, URL validation, browser launch helpers, registry state, and persistence.
- `execute_actions` performs action value rendering, selector rendering, guard branching, CDP execution dispatch, upload screenshot collection, error publication, and loop control.
- `execute_action_once` handles navigation, pauses, reload, page-loaded waits, selector actions, file upload path safety, keyboard input, screenshots, network wait approximations, request wait approximations, and cache clearing.
- Upload file handling correctly rejects absolute paths and path traversal before resolving against upload dir.
- Action error publishing is tied directly to websocket event emission.
- `PlaceholderCallbacks` combines runtime state mutation, websocket publishing, result persistence, upload persistence, and step lifecycle events.

Architecture classification:

- HIGH localized risk.
- This is the actual workflow engine and browser adapter in one file.
- The code contains useful safety checks, but architecture boundaries are dense: rendering, browser automation, upload handling, persistence callbacks, websocket events, and job-state transitions are tightly colocated.

### `zen-server/src/features/job/repo.rs`

Observed:

- `JobRepository` owns SQLite pool and a `OnceCell` schema readiness guard.
- `ensure_schema` invokes `rename_job_id_columns`, which narrowly handles legacy `request_id` to `job_id` column migration for `jobs` and `job_results`.
- Repository methods persist summaries, insert results, read/update job logs JSON, clear results, page jobs, fetch summaries, fetch paged results, and check successful result existence.
- Query building uses `QueryBuilder` for list filters and explicit selected columns.
- Mapping helpers parse JSON vectors, task states, and RFC3339 datetimes.

Architecture classification:

- MEDIUM risk.
- Most code is persistence-focused and aligns with the repo boundary.
- The runtime schema compatibility probe is documented and narrow, but it is still a repository-side schema mutation and should remain a bounded exception only.

### `zen-server/src/features/periodic/service.rs`

Observed:

- `PeriodicService` owns flow, system, dataset, and job service dependencies.
- `generate_due_cycles_at` scans fill flows, resolves monthly/yearly period keys, loads systems, creates generated datasets when needed, creates pending template jobs, and records summary counts.
- Period key generation uses local time and maps monthly reports to previous month and yearly reports to previous year.
- Tests cover monthly and yearly period-key behavior.

Architecture classification:

- MEDIUM risk.
- This service is intentionally orchestration-heavy, but it stays compact and readable.
- Its main coupling is vertical: flow -> system -> dataset -> job.

## PASS 2 — Cross-Module Consistency

### Documentation vs implementation

Consistent:

- Root and docs consistently define `zen-server` + `zen-web` as a protected layout.
- Backend feature layout matches the documented `features/<feature>/` pattern.
- SQLite-first runtime and local upload/static/log layout match README and backend guide.
- `overview` remains documented as a read-only projection exception.
- The narrow `JobRepository::rename_job_id_columns` exception is documented in backend guide and visible in implementation.

Inconsistent or pressured:

- `service.rs` is documented as orchestration, validation, and transactions, but `job/service.rs` also owns dataset row-shape rules, log serialization, period filtering, runtime reconciliation, and execution planning.
- `scheduler.rs` is named as scheduler, but it performs resource setup, data conversion, account secret resolution, browser dispatch, persistence, and websocket finalization.
- `executor.rs` is named as executor, but it combines browser adapter, DSL action interpreter, rendering failure reporting, uploads, persistence callbacks, websocket events, and runtime state mutation.
- `app.rs` is documented as router, middleware, and state assembly; implementation matches this, but the dependency graph is dense enough to be a recurring Web/Rust risk point.

### Consistency with Rustzen Architecture v1

Aligned:

- Local-first SQLite runtime.
- Explicit Axum/Tokio runtime.
- Feature-local `api/service/repo/types` shape.
- Direct API-to-service and service-to-repo dependency direction.
- Documentation-as-control-surface for AI-assisted development.

Pressured:

- Job workflow violates the spirit, though not always the letter, of narrow service boundaries.
- Browser automation path creates a second architecture axis that does not fit simple CRUD layering.
- Runtime schema compatibility is controlled, but it is still a rule exception.

## PASS 3 — Unified Architecture Synthesis

### Risk rating

`rustzen-report` is classified as **MEDIUM-HIGH risk**.

Reason:

- Repository-level layout and documentation are strong.
- Startup and runtime model are explicit.
- Risk is concentrated in the report job workflow: `job/service.rs`, `job/scheduler.rs`, and `job/executor.rs` form a dense chain spanning planning, runtime, browser automation, persistence, websocket events, uploads, and period generation.

### Per-repo consistency check

PASS:

- Protected `zen-server` / `zen-web` layout is explicit and should not be treated as generic migration debt.
- Runtime path model is clear.
- SQLite-first database path is consistent.
- Docs define concrete backend rules.
- Narrow legacy DB compatibility exception is documented.

WATCH:

- `AppState` composition graph is large.
- `JobService` has too many workflow responsibilities.
- `Scheduler` performs setup and terminal event handling beyond scheduling.
- `Executor` is both workflow engine and browser adapter.
- Browser automation introduces domain-specific architecture pressure not covered by CRUD-only rules.

### Unified Rustzen architecture rules update

Add the following rule to Rustzen Architecture v1 audit notes:

> For browser-driven automation repositories, the standard `api -> service -> repo` model is necessary but not sufficient. The workflow engine must be audited as a first-class boundary. Service files may orchestrate jobs, but DSL interpretation, browser/CDP execution, upload capture, websocket event emission, runtime state transitions, and persistence callbacks must not silently collapse into one unbounded execution module.

Additional clarification:

> A protected legacy/current layout can pass repository-level standardization while still carrying MEDIUM-HIGH risk in one feature path. Rustzen audit must therefore score root layout and workflow density separately.

## Blog Series Candidate

### Title

Designing a Local-First Report Automation System Without Losing Workflow Boundaries

### Type

Technical article.

### Draft narrative

A local-first report automation tool has a different shape from a normal admin system. It needs users, systems, accounts, datasets, templates, schedules, job history, uploads, and live status updates, but its real center is not CRUD. Its center is the workflow path that turns a report definition into a browser-executed task.

`rustzen-report` shows the advantage and risk of that design. The repository keeps a clear runtime model: SQLite under `data/`, uploads under `data/uploads/`, static frontend output under `web/dist/`, and logs under `logs/`. It also documents the protected `zen-server` and `zen-web` layout, so the system is not accidentally forced into a different standard layout.

The architectural pressure appears lower in the repository root and higher inside the job path. `JobService` prepares requests, resolves datasets and accounts, creates runtime state, persists summaries, and starts jobs. The scheduler then resolves templates, records, accounts, secrets, and terminal websocket events. The executor interprets actions, drives the browser, captures screenshots, publishes progress, persists results, and updates runtime state.

That is the key lesson: a browser-driven automation product needs a workflow boundary, not just a service boundary. CRUD layering keeps data access sane, but it does not automatically control the complexity of execution logic. The report engine must be treated as its own architectural surface.

For Rustzen, this confirms a broader rule. Local-first systems can stay simple at the infrastructure layer and still become dense in one workflow. The audit model therefore needs two scores: one for repository layout, and one for the highest-pressure execution path.
