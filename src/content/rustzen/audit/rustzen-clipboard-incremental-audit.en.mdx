## PASS 1 — File-Level Inspection

### `README.md`

Observed:

- Product positioning is explicit: local-first clipboard history for macOS.
- The repository is GUI-first and intentionally has no terminal workflow.
- Product scope covers text/image capture, local history, search, pin/delete, preview, and restore.
- Non-goals explicitly exclude CLI, cloud backend, public website, cloud sync, and hidden upload.

Architecture classification:

- Product boundary is clear.
- Privacy posture is aligned with Rustzen local-first rules.
- This repo should not be judged by CLI-first runtime rules; it is a GUI-first local app with a local backend bridge.

### `docs/10-architecture-design.md`

Observed:

- The documented flow is `system clipboard -> zen-core -> local storage -> zen-gui -> explicit restore`.
- `zen-core` is documented as capture, normalization, history model, search/filtering, pin/delete/retention, restore, privacy rules, and local HTTP backend.
- `zen-gui` is documented as desktop window, app state, history list, search/filter, previews, interactions, and settings.
- The doc requires GUI to call `zen-core` instead of duplicating core logic.

Architecture classification:

- The intended boundary is clear but broad.
- `zen-core` is not only domain logic; it is domain + storage + HTTP API + local backend runtime.
- This is coherent for a compact product, but it conflicts with strict Rustzen v1 separation unless treated as a transitional compact-core architecture.

### `docs/20-project-map.md`

Observed:

- Repository contains `docs`, `zen-core`, `zen-gui`, `AGENTS.md`, and `README.md`.
- `zen-core/src` contains `clipboard.rs`, `config.rs`, `error.rs`, `http.rs`, `lib.rs`, `main.rs`, `models.rs`, `server.rs`, and `storage.rs`.
- Removed scope includes `zen-cli`, `zen-cloud`, `zen-website`, `.github`, and old history/report/review docs.

Architecture classification:

- The repo has already been reduced to a focused GUI + local backend product.
- Current risks are therefore inside module boundaries, not product scope sprawl.

### `zen-core/Cargo.toml`

Observed:

- `zen-core` combines clipboard access (`arboard`, `objc2-app-kit`), image normalization, SQLite (`sqlx`), local HTTP (`axum`, `tower`, `tower-http`), async runtime (`tokio`), tracing, serialization, hashing, and OS directories.
- There is no visible internal crate split between domain, storage, HTTP handlers, runtime, and platform adapters.

Architecture classification:

- `zen-core` acts as a combined product backend crate.
- Dependency shape confirms `zen-core` is not pure core/domain.
- Risk level: HIGH until storage, capture, HTTP, and runtime boundaries are separated or explicitly accepted as a compact-core exception.

### `zen-core/src/lib.rs`

Observed:

- Public modules are exported directly: `clipboard`, `config`, `error`, `http`, `models`, `server`, and `storage`.

Architecture classification:

- Public surface is convenient but flat.
- Internal adapters and HTTP/runtime modules are visible at the crate root, weakening boundary discipline.

### `zen-core/src/main.rs`

Observed:

- Binary entry is minimal: async `main` delegates to `zen_core::server::run()`.

Architecture classification:

- Runtime entry is thin and acceptable.
- The risk is not the binary entrypoint; it is what `server::run()` owns.

## PASS 2 — Cross-Module Consistency

### Product scope

Finding:

- Documentation and code agree that the current product is GUI-first with no CLI or cloud backend.
- The removed-scope list is consistent with the current repository layout.

### Core boundary

Finding:

- `zen-core` is documented as the owner of capture, local storage, GUI API bridge, restore, retention, and privacy.
- Implementation matches that broad boundary, but strict Rustzen v1 treats those as separate responsibility groups.
- `storage.rs` and `clipboard.rs` are the main pressure points.

### HTTP vs Tauri command boundary

Finding:

- The GUI uses a local HTTP backend for most history/settings operations.
- Tauri command `paste_history_entry` directly creates `AppStorage` and calls core restore/storage methods.
- This creates two interaction paths into core state: HTTP routes and Tauri commands.
- The split is operationally understandable for paste automation, but it weakens the single application-service boundary.

### Storage vs workflow boundary

Finding:

- Storage owns database persistence, settings, retention policy, ignore-next state, file cleanup, and history ranking.
- Clipboard owns capture/restore but directly triggers storage writes and retention application.
- No narrow service layer exists between capture/HTTP/Tauri and persistence.

### Privacy and safety boundary

Finding:

- README explicitly prohibits hidden upload and cloud sync.
- Code remains local-first and validates image/icon paths through app-owned canonicalization.
- Sensitive text detection exists but is a simple keyword heuristic and lives inside `clipboard.rs`.

## PASS 3 — Unified Architecture Synthesis

### Confirmed repo status

`rustzen-clipboard` remains HIGH risk.

The repo is product-focused and smaller than `rustzen-clear`, but it has two confirmed high-pressure modules:

- `zen-core/src/storage.rs`: persistence + settings + retention + file cleanup + ranking + ignore-next state.
- `zen-core/src/clipboard.rs`: capture loop + platform adapter + image normalization + source app detection + restore + retention trigger.

A secondary risk exists in:

- `zen-gui/src-tauri/src/main.rs`: window lifecycle + tray/global shortcut + backend startup + direct storage-backed paste orchestration.

### Standard refinement from this pass

For GUI-first local Rustzen products:

- `zen-core` may expose a compact local backend, but capture, storage, HTTP handlers, runtime bootstrap, and GUI/Tauri automation must remain auditable as separate responsibility groups.
- If both HTTP and Tauri commands invoke product workflows, they must share the same application-service boundary rather than opening independent storage paths.
- Retention, ranking, ignore-next capture, and file cleanup are product policies; they should not be hidden inside generic persistence APIs.
- Platform adapters such as pasteboard, `osascript`, `plutil`, and `sips` must remain isolated from domain models and storage rules.
- Local-only privacy claims must be preserved by making external-network and cloud-sync absence explicit in dependencies and documentation.

## Audit Delta

This pass does not change the global priority order, but it sharpens `rustzen-clipboard` from a generic “storage + capture coupling” issue into three precise boundaries:

1. `storage.rs` is a policy-heavy repository module, not just DB access.
2. `clipboard.rs` is a capture/restore/platform engine, not just clipboard abstraction.
3. `zen-gui/src-tauri/src/main.rs` creates a second workflow entry path for paste automation.

The unified rule is refined as follows:

- A GUI-first Rustzen app may omit CLI, but it still needs a single product workflow boundary.
- Local HTTP handlers and Tauri commands must not diverge into separate orchestration paths.
- Retention and privacy-sensitive clipboard behavior must be first-class workflow policy, not incidental storage or adapter behavior.

### Article fit

This repo has enough meaningful architecture insight for a technical article, but the article should remain technical rather than promotional.

### Core narrative

Rustzen Clipboard is intentionally smaller than a cloud-backed productivity system: no CLI, no cloud sync, no public website, and no hidden upload path. That narrow product scope is a strength. The architectural challenge is different: a local-first app can still become hard to evolve when capture, storage, retention, HTTP handlers, and Tauri automation all converge into the same crate without a single workflow boundary.
