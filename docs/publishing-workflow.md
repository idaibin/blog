# 跨平台内容发布流程

本文档定义 `blog` 文章从事实整理、写作审核到跨平台发布和发布后检查的默认流程。目标是让博客成为稳定的内容源，同时保留人工审核和各平台必要的表达差异。

## 基本原则

- `blog` 是中文和英文长文的内容源，也是对外链接的默认落点。
- 先验证事实，再优化表达。Git 记录、项目现状和真实运行经历用于核对内容，不直接写成提交时间线。
- 博客草稿与外部发布分开处理。用户审核通过以前，不更新任何外部平台。
- 国内平台使用中文内容，海外平台使用英文内容。
- 跨平台发布是适配，不是复制。保留同一事实主线，根据平台调整长度、开头、链接和互动方式。
- 任何发布动作都需要明确授权；填写草稿不等于批准最终发布。

## 标准流程

### 1. 明确文章范围

先确认文章要回答的问题、目标读者和不应公开的内容。涉及项目经历时，核对当前仓库、Git 历史、已有文章和用户补充的运行事实，区分以下内容：

- 已经完成并可以验证的现状；
- 尝试过但已经放弃或调整的方案；
- 真实运行中遇到的问题；
- 尚未完成的计划和方向。

不要把计划写成已经完成的结果，也不要为了让过程更顺畅而补写没有证据的原因、指标或结论。

### 2. 先完成博客中文稿

中文稿先进入 `src/content/blog/`，沿用现有 frontmatter、分类和路由规则。写作重点是形成一条清楚的事实与判断主线，而不是罗列功能或逐条复述提交记录。

需要外部写作审查时，审查材料只保留完成任务所需的事实、边界和正文。审查意见必须回到原始证据中验证，再决定是否采用。

### 3. 保留用户审核门槛

完成博客草稿后先交给用户审核。根据反馈继续修改博客源稿，直到文章内容、语气、公开边界和后续计划都得到确认。

此阶段不执行以下操作：

- 更新或发布掘金文章；
- 发布 DEV、X、Reddit 或小红书内容；
- 把外部平台的草稿状态当成最终发布授权。

### 4. 准备英文版本

英文版使用与中文稿相同的 slug 和事实主线，但按英文技术文章的表达习惯重新组织句子，不做逐句直译。涉及中文平台历史时补充必要上下文，避免海外读者需要先了解国内平台或项目背景。

发布前检查中英文版本是否保持以下内容一致：

- 项目定位和当前边界；
- 关键技术与角色设计；
- 运行问题及其状态；
- 已完成工作与后续计划的区别；
- 仓库和博客链接。

### 5. 验证并交付博客

在独立分支或干净 worktree 中完成内容改动，避免混入现有草稿和无关修改。提交前执行：

```bash
npm run build
npm run astro -- check
```

只暂存本次文章和相关文档，检查 staged diff 后提交。按仓库规则 squash 合并到 `main`，推送后确认本地 `main`、`origin/main` 和预期提交一致。

### 6. 验证线上博客

发布外部平台以前，分别打开中文和英文页面，检查：

- 页面返回正常，标题、描述、日期和正文正确；
- 中文地址使用 `/zh/` 路径，英文地址使用默认路径；
- 中英文切换和 canonical URL 正确；
- GitHub、历史文章及其他外部链接可用；
- 分享卡片的标题、描述和图片符合预期。

## 平台适配

### 掘金

使用完整中文技术文章。可以根据掘金读者补充简短背景，但事实主线应与博客中文稿一致。文章中的延伸阅读优先链接中文博客，项目入口链接 GitHub。

### 小红书

使用中文短内容，围绕一个明确变化、问题或经验展开。减少完整架构说明和长功能列表，优先使用短段落、清楚的小标题和可独立阅读的结论。需要多图时，再把文章拆成卡片结构。

### DEV

使用完整英文文章。博客英文版作为内容源；如果 DEV 支持 canonical URL，应填写英文博客地址，避免形成两个互相竞争的长文来源。

### X

使用英文短帖或自然 Thread。首条先说明项目、变化和读者为什么值得继续看，再决定是否附 GitHub 或英文博客链接。只有在长 Thread 确实需要阅读顺序时才使用 `1/4` 一类编号，不把编号当成固定格式。

### Reddit

使用英文讨论型内容，说明实际问题、方案取舍和希望讨论的问题，避免只放项目宣传和链接。账号或社区门槛不满足时，先发布到个人主页并参与相关技术讨论，不反复向已经移除内容的社区投稿。

发布后额外检查：

- 帖子和个人主页没有被错误标记为 NSFW；
- 帖子是否被平台过滤或版主移除；
- 社区规则、Karma 和账号年龄是否满足要求；
- 链接指向英文博客或 GitHub，而不是中文页面。

## 发布后记录

每轮发布保留一个简短状态表，避免把“已填写”“已提交审核”和“已公开”混为一谈：

| 平台 | 语言 | 形式 | 状态 | 地址或说明 |
| --- | --- | --- | --- | --- |
| Blog | 中文 / 英文 | 完整文章 | 草稿 / 已发布 | 中英文页面 |
| 掘金 | 中文 | 完整文章 | 草稿 / 已发布 | 文章地址 |
| 小红书 | 中文 | 短文 / 卡片 | 草稿 / 已发布 | 笔记地址 |
| DEV | 英文 | 完整文章 | 草稿 / 已发布 | 文章地址 |
| X | 英文 | 单帖 / Thread | 草稿 / 已发布 | 帖子地址 |
| Reddit | 英文 | 个人帖 / 社区讨论 | 草稿 / 已发布 / 被移除 | 帖子地址和社区状态 |

最后检查各平台的实际公开页面，而不是只依据编辑器或发布按钮的状态。记录异常原因，例如链接错误、语言版本错误、NSFW 标记、社区过滤或审核移除，作为下一轮发布时的检查项。
